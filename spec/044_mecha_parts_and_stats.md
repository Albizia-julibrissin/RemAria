# Spec: メカパーツ・部位・ステータス計算

`docs/024_mecha_design.md` に基づき、メカのパーツ部位・マスタデータ構造・フレーム補正の持ち方・ステータス計算ルールを実装可能な形で定義する。

------------------------------------------------------------------------

## 0. 依存・横断

### 0.1 依存する spec

- **010_auth**：ログイン済み前提の画面・API が存在する。
- **015_protagonist_creation**：主人公が存在する。
- **020_test_battle**：戦闘でメカを出撃させる場合の土台。
- **038_battle_skills_and_effects**：メカスキルは同一 `Skill` を category で区別して利用する想定。

### 0.2 提供する API / 利用者

| API（想定） | 用途 | 呼び出し元 |
|-------------|------|------------|
| getMechaPartTypes | パーツ種別一覧（装備画面・クラフト一覧用） | 装備画面・工業画面 |
| computeMechaBaseStats | 装備パーツからメカの基礎ステータスを算出 | 戦闘計算・ステータス表示 |

※ 装備の着脱・クラフト結果のインスタンス管理は別 spec（装備サイクル・在庫）に委ねる。

------------------------------------------------------------------------

## 1. 目的

- メカの **6 部位** と「フレームは倍率・他は加算」というルールを正本化する。
- **フレームの補正**を、スキーマ変更に強い形で持つ（本 spec では JSON で定義）。
- パーツ種別マスタとメカ用スキル参照を定義し、戦闘時の「使えるスキル一覧」と「基礎ステ算出」の契約を定める。

------------------------------------------------------------------------

## 2. 用語

- **メカ**：`Character.category = 'mech'` のユニット。主人公・仲間と異なり装備で **基礎ステータス** が決まる。
- **部位（スロット）**：Frame / Core / Head / RightArm / LeftArm / Legs の 6 種。1 メカあたり各 1 スロット。
- **パーツ種別（MechaPartType）**：部位ごとのマスタ。1 種別は 1 部位に属し、フラット加算値またはフレームのときは倍率のみを持つ。
- **フレーム補正（statRates）**：部位が Frame のときのみ使用。基礎ステータスごとの **倍率**（例: STR 1.2, DEX 0.8）。キーなしは 1.0 扱い。
- **基礎ステータス**：STR, INT, VIT, WIS, DEX, AGI, LUK（7 種）＋ CAP。`docs/10_battle_status.csv`・`docs/16_base_stats_alignment.md` 準拠。メカは装備なし時は **全 10**（CAP は別途定義可）。

------------------------------------------------------------------------

## 3. 部位（スロット）定義

| 定数 | 説明 | ステータス役割 | スキル傾向 |
|------|------|----------------|------------|
| **Frame** | フレーム | 倍率のみ（statRates） | なし |
| **Core** | コア | フラット加算 | 任意 |
| **Head** | 頭部 | フラット加算 | デバフ系 |
| **RightArm** | 右腕 | フラット加算 | 攻撃 1 つ |
| **LeftArm** | 左腕 | フラット加算 | 攻撃 1 つ |
| **Legs** | 脚部 | フラット加算 | バフ系 |

- コード上は enum または文字列定数で `frame` / `core` / `head` / `rightArm` / `leftArm` / `legs` を定義する。
- スキル傾向は設計ガイドであり、データ上は「1 パーツ複数スキル」を許容する（初期は 1:1）。

------------------------------------------------------------------------

## 4. フレーム補正の持ち方（statRates）

フレームのみ「加算」ではなく「倍率」を持つため、次のいずれかで保持する。

### 4.1 採用案：JSON で保持（推奨）

- **フィールド名**：`statRates`（型: Json, nullable）
- **使用する部位**：`slot = 'frame'` のパーツ種別のみ。他は null。
- **形式**：キー = 基礎ステータス名（`STR` / `INT` / `VIT` / `WIS` / `DEX` / `AGI` / `LUK`）。必要に応じて `CAP` も含めてよい。値 = 倍率（数値）。小数可（例: 1.2, 0.8）。
- **解釈**：
  - キーが存在する基礎ステにはその倍率をかける。
  - **キーが無い**基礎ステは **1.0** として扱う（省略可能）。
- **例**：`{ "STR": 1.2, "DEX": 0.8 }` → STR は 1.2 倍、DEX は 0.8 倍、それ以外は 1.0 倍。

**利点**：将来基礎ステを増やしてもスキーマ変更不要。フレームだけの例外を 1 フィールドに閉じ込められる。

### 4.2 代替案：カラムで保持

- フレーム用に `strRate`, `intRate`, `vitRate`, `wisRate`, `dexRate`, `agiRate`, `lukRate`（Decimal, default 1.0）を持つ。
- 利点：型安全・クエリしやすい。欠点：ステータス追加時にマイグレーションが必要。

※ 本 spec では **4.1（JSON）** を正本とする。実装時に 4.2 を選ぶ場合は、本 spec の「statRates の解釈」ルールをカラム読みに置き換える。

------------------------------------------------------------------------

## 5. データ構造

### 5.1 パーツ種別マスタ（MechaPartType）

| 項目 | 型 | 説明 |
|------|-----|------|
| id | String (cuid) | 主キー |
| slot | String | 上記 6 種のいずれか（frame / core / head / rightArm / leftArm / legs） |
| name | String | 表示名（例: ガンアーム、重装脚） |
| statRates | Json? | **slot=frame のときのみ**使用。倍率オブジェクト（4.1 参照）。他は null |
| strAdd, intAdd, vitAdd, wisAdd, dexAdd, agiAdd, lukAdd, capAdd | Int? | **slot≠frame のとき**使用。フラット加算。未指定は 0 扱い |
| createdAt / updatedAt | DateTime | 通常監査 |

- **スキル**：1 パーツ種別が 0 個以上のスキルを参照する。中間テーブル `MechaPartTypeSkill (mechaPartTypeId, skillId, orderIndex)` または Json 配列 `skillIds` のいずれかで保持。**初期は 1:1、拡張で複数可**。
- 同一 slot に複数種別が存在可能（例: ガンアーム / 重ガンアーム はどちらも rightArm）。

### 5.2 装備情報（実装フェーズで定義）

- メカ（Character, category=mech）が「どのパーツ種別（またはクラフト結果インスタンス）をどの部位に装着しているか」を保持するテーブルを別 spec または本 spec の拡張で定義する。
- 本 spec では「装着済みパーツの MechaPartType の一覧が与えられたとき、基礎ステと使えるスキルが決まる」という契約のみ定める。

### 5.3 メカスキルと Skill テーブル

- メカ専用スキルは **既存 `Skill` テーブル** に `category = 'mecha'`（または `battle_mecha` 等）で登録する。
- effectType や戦闘解決は spec/038・docs/042 に従い、キャラ用と共通の戦闘エンジンで扱う。

------------------------------------------------------------------------

## 6. ルール

### 6.1 メカの基礎ステータス計算

1. **ベース**  
   装備なし、またはその部位に何も装着していない場合は、その部位の寄与は 0（フレームは倍率 1.0 扱い）。  
   メカ全体で **1 つも装備していない** ときは、全基礎ステ **10**（CAP は仕様で 60 等と定義）。

2. **フラット加算の合算**  
   slot が frame 以外の全装着パーツについて、`strAdd` ～ `capAdd` をそれぞれ合算する。  
   → 仮の基礎ステ `rawStr`, `rawInt`, … が決まる（ベース 10 に上記を加算。装備なしなら 10 のまま）。

3. **フレーム倍率の適用**  
   slot=frame の装着パーツが 1 つあるとき、そのパーツの `statRates` を読む。  
   - 各基礎ステ `s` について: `final[s] = round(raw[s] * (statRates[s] ?? 1.0))`（または truncate。仕様で統一すること）。  
   - フレームを装着していない場合は `final[s] = raw[s]`。

4. **最終値**  
   `final` がメカの基礎ステータス。ここから `docs/10_battle_status.csv` に従い派生ステ（HP・攻撃力・速度など）を算出する。

### 6.2 メカが使えるスキル

- 装着している全パーツ種別が持つ `skillId`（または skillIds）を **重複除いて** 集めたものが、そのメカの「使用可能スキル一覧」。
- 作戦スロット（spec/039）で選択できるスキルはこの一覧に限定する。メカは CharacterSkill で習得は持たない。

### 6.3 フレーム以外の statRates

- slot が frame 以外のパーツ種別では `statRates` は **参照しない**。フラット加算のみ。

------------------------------------------------------------------------

## 7. 処理フロー（概要）

### computeMechaBaseStats（概念）

1. 入力: メカの 6 部位に対するパーツ種別（またはインスタンス→種別）の一覧。
2. ベースを 10 で初期化。フレーム以外の部位から strAdd ～ capAdd を合算し raw を求める。
3. フレーム部位の statRates を取得。各 raw に倍率をかけ final を求める。
4. 出力: `{ STR, INT, VIT, WIS, DEX, AGI, LUK, CAP }`（整数）。

### 戦闘時の利用

- メカのターン・ダメージ計算時は、上記で求めた基礎ステを `derived-stats` 等に渡し、既存の二次解釈で派生ステを得る。

------------------------------------------------------------------------

## 8. 永続化データ

### 8.1 永続化するデータ

- **MechaPartType**：部位（slot）、名前、statRates（frame のみ）、strAdd～capAdd（frame 以外）、およびスキル参照（中間テーブルまたは Json）。
- **Skill**：既存。メカ用は category で区別。
- 装備関係（どのメカがどのパーツをどの部位に装着しているか）は、装備・在庫 spec に合わせてテーブル設計する。

### 8.2 保存しないデータ

- 計算結果の「メカの現在の基礎ステータス」は、装備から都度計算する方針でよい（キャラの装備で戦闘用最終値を保存しないのと同様）。必要ならキャッシュは実装側で行う。

------------------------------------------------------------------------

## 9. 例（最低 3 ケース）

### ケース 1：装備なし

- 入力: 全部位なし。
- 期待: 全基礎ステ 10（CAP は仕様どおり）。

### ケース 2：フレームのみ（倍率のみ）

- 入力: Frame に statRates = `{ "STR": 1.2, "DEX": 0.8 }` のパーツのみ装着。他部位なし。
- raw: 全 10。final: STR=12, DEX=8, 他 10。
- 期待: STR=12, INT=10, VIT=10, WIS=10, DEX=8, AGI=10, LUK=10。CAP は statRates にキーが無ければ 10 のまま（または CAP は別規定）。

### ケース 3：フラット＋フレーム

- 入力: Core が strAdd=50, dexAdd=30。他 0。Frame が statRates = `{ "STR": 1.2 }`。
- raw: STR=60, DEX=40, 他 10。
- final: STR=72, DEX=40, 他 10。
- 期待: STR=72, DEX=40, その他 10。

------------------------------------------------------------------------

## 10. エラー条件 / NG

- slot が frame なのに statRates が null のときは、倍率を全て 1.0 として扱う（フォールバック）。
- statRates に不正なキー（存在しない基礎ステ名）が含まれる場合は無視する。
- 同一部位に複数装着は許可しない（1 メカ 1 部位 1 パーツ）。

------------------------------------------------------------------------

## 11. テスト観点

- 装備なしで全 10 になること。
- フレームのみで statRates のキーに対応するステだけ倍率がかかり、他が 1.0 であること。
- フラット加算のみ（フレームなし）で raw がベース 10＋加算になること。
- フラット加算のあとフレーム倍率をかけた結果が、手計算と一致すること。
