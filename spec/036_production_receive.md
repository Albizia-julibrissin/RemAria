# Spec: 生産受け取り（全設備一括）

`docs/019_production_receive.md` に基づき、**全設備一括受け取り**の API・在庫スナップショット・処理順・lastProducedAt 更新（在庫不足時は受け取り実行日時に更新）を実装する。

------------------------------------------------------------------------

## 0. 依存・横断

### 0.1 依存する spec

- **010_auth**：セッションが有効であること。受け取り API は getSession を前提とする。
- **035_initial_area_facilities**：FacilityInstance・Recipe・RecipeInput・Item・User の工業まわりが存在すること。在庫（UserInventory）は本 spec で必要とするため、035 または別 spec で定義する。
- **docs/019**：計算ルール・用語・lastProducedAt 更新（在庫不足時含む）は 019 に準拠する。

### 0.2 提供する API / 利用者

| API | 用途 | 呼び出し元 |
|-----|------|-----------|
| receiveProduction | 全設備の生産を一括で受け取り、在庫増減・各設備の lastProducedAt を更新する | 工業画面の「受け取る」ボタン |
| （設備配置変更 API 内で receiveProduction を呼ぶ） | 強制受け取り：変更前の設備構成で 1 回受け取り実行後、設備の追加・撤去・入れ替えを反映 | 設備の建設・解体・入れ替え確定時 |

------------------------------------------------------------------------

## 1. 目的

- プレイヤーが「工業の生産物を受け取る」ボタンを押したとき、**全設備**について経過時間とレシピからサイクル数を算出し、在庫を消費・付与して各設備の **lastProducedAt** を更新する。
- **在庫スナップショット**は受け取り実行時点の在庫 1 回取得。同一資源を複数設備が使う場合は**設備を確定順（例：FacilityInstance.id 昇順）**で処理し、**作業用コピー**から先に並んだ設備の消費分を差し引いてから次を計算する。
- **在庫不足**で実行サイクル数が時間ベースより少なかった設備は、lastProducedAt を**受け取り実行日時**に更新し、未生産の経過を繰り越さない。
- **設備の配置変更時**は、変更前の設備構成で上記受け取りを 1 回実行してから、設備の追加・撤去・入れ替えを DB に反映する（強制受け取り）。

------------------------------------------------------------------------

## 2. 用語

- **受け取り**：receiveProduction の 1 回実行。全設備を対象にスナップショット→計算→在庫反映・lastProducedAt 更新をトランザクションで行う。
- **スナップショット**：受け取り開始時に取得した UserInventory の全件（userId, itemId, quantity）。作業用コピーはこのコピーを更新しながら各設備の入力可能回数を計算する。
- **時間ベースのサイクル数**：floor( min(経過分, 1440) / cycleMinutes )。経過は lastProducedAt（NULL なら User.createdAt または FacilityInstance.createdAt）から受け取り実行日時まで。
- **実行サイクル数**：min( 時間ベースのサイクル数, 作業用コピーで実行可能な回数 )。入力なし設備は時間ベースのまま。

------------------------------------------------------------------------

## 3. 入力・出力

### 3.1 receiveProduction の入力

- **セッション**：getSession() から得た userId。入力パラメータは不要（「受け取る」ボタン 1 クリックで実行）。

### 3.2 receiveProduction の出力

- **成功時**：受け取った生産物のサマリ（設備ごとの出力増加・入力消費の要約）、および更新後の lastProducedAt 情報が必要なら返す。クライアントでは「〇〇を N 個受け取りました」等を表示してよい。
- **強制受け取り**の場合は、同一処理の返却に「区画整理のため、一度生産資源を回収しました」などのメッセージを含め、クライアントで表示する。

------------------------------------------------------------------------

## 4. 永続化データ（本 spec で利用するもの）

### 4.1 在庫（UserInventory）

| カラム | 型 | 説明 |
|--------|-----|------|
| userId | String, FK→User | 所有者 |
| itemId | String, FK→Item | アイテム |
| quantity | Int | 所持数（0 以上） |

- 受け取り開始時に**全件読み取り**してスナップショットとする。計算後に**消費・付与を一括で更新**する（トランザクション内）。

### 4.2 FacilityInstance（035 と共通）

- **lastProducedAt**（DateTime, NULL 可）：019 に従い、受け取り後に次で更新する。
  - 実行サイクル数 ＝ 時間ベースのサイクル数 → **lastProducedAt + 実行サイクル数 × cycleMinutes**
  - 実行サイクル数 ＜ 時間ベースのサイクル数（在庫不足で入力制限された）→ **受け取り実行日時（now）**

### 4.3 Recipe / RecipeInput / Item（035 と共通）

- 各設備の FacilityType に紐づく Recipe の cycleMinutes、outputItemId、outputAmount、および RecipeInput（itemId, amount）を参照する。

------------------------------------------------------------------------

## 5. 処理フロー（receiveProduction）

1. **トランザクション開始**。必要ならユーザー在庫でロック（二重押し防止）。
2. **スナップショット取得**：userId に紐づく UserInventory を全件取得。**作業用コピー**をメモリ上に構築（itemId → quantity のマップでよい）。
3. **User 取得**：lastProducedAt が NULL の強制配置設備で「経過の開始時刻」のフォールバックに使うため、User を 1 件取得し **createdAt** を保持する。
4. **設備一覧取得**：userId に紐づく FacilityInstance を**確定順（id 昇順）**で取得。各 FacilityInstance に FacilityType → Recipe, RecipeInput を join。
5. **現在時刻**を 1 回取得（受け取り実行日時 = now）。全設備で共通に使う。
6. **設備を順に処理**（id 昇順）：
   - 経過の開始時刻 = lastProducedAt ?? (FacilityInstance.isForced ? User.createdAt : FacilityInstance.createdAt)（019 に従う）。
   - 経過分 = now - 経過の開始時刻（分）。キャップ 1440 分。
   - 時間ベースのサイクル数 = floor(経過分 / cycleMinutes)。
   - RecipeInput がある場合：作業用コピーから各入力の「所持数 / 1回あたり消費数」の floor の最小値を取り、実行可能回数とする。実行サイクル数 = min(時間ベースのサイクル数, 実行可能回数)。この設備の入力消費分を作業用コピーから減算する。
   - RecipeInput が無い場合：実行サイクル数 = 時間ベースのサイクル数。
   - この設備の「消費する入力」「付与する出力」をリストに積む。lastProducedAt の更新値も記録（実行サイクル数 ＝ 時間ベースなら 開始時刻 + 実行サイクル数×cycleMinutes、そうでなければ now）。
7. **在庫反映**：リストに積んだ全入力消費・全出力付与を UserInventory に一括反映（存在しなければ INSERT、あれば quantity を加減算）。
8. **lastProducedAt 更新**：各 FacilityInstance の lastProducedAt を 6 で求めた値に更新。
9. **トランザクションコミット**。
10. 返却：受け取った生産物のサマリ（および強制受け取り時はメッセージ）。

------------------------------------------------------------------------

## 6. 強制受け取り（設備配置変更時）

- 設備の**追加・撤去・入れ替え**を確定する API の処理順序は以下とする。
  1. **変更前の設備構成**で **receiveProduction(userId)** を 1 回呼ぶ（変更はまだ DB に反映しない）。
  2. 返却に「区画整理のため、一度生産資源を回収しました」を含め、クライアントで表示する。
  3. そのあとで設備の追加・撤去・入れ替えを DB に反映する（新規 FacilityInstance の lastProducedAt は NULL のまま作成してよい）。
- これにより、新規追加設備はその受け取りの対象外となり、次回受け取りではその設備の経過は配置日時（createdAt）から積み上がる。

------------------------------------------------------------------------

## 7. 初回の lastProducedAt フォールバック

- lastProducedAt が NULL のときの「経過の開始時刻」：
  - **isForced === true** の設備（強制配置 5）→ **User.createdAt**
  - **isForced === false** の設備 → **FacilityInstance.createdAt**

------------------------------------------------------------------------

## 8. テスト観点

- 受け取り実行後、時間ベースで実行した分だけ lastProducedAt が「前回 + 実行サイクル数×周期」で進んでいること。
- 在庫不足で 0 サイクルだった設備は、lastProducedAt が**受け取り実行日時**に更新されていること。次回受け取りで経過が 0 から始まること。
- 同一資源を消費する設備が 2 つあるとき、処理順で先の設備が在庫を消費し、後の設備は残り在庫で実行可能回数が制限されること。
- 強制受け取り時、変更前の設備のみが受け取り対象であり、変更後に追加した設備の lastProducedAt は NULL のままであること。
- 二重押しや同時リクエストで在庫・lastProducedAt が二重に更新されないこと（ロックまたは冪等設計）。

------------------------------------------------------------------------

## 9. エラー条件

- セッションが無い：401 またはログイン促し。
- UserInventory が存在しない／スキーマが未整備：実装時に在庫 spec と合わせて用意する。本 spec では「userId + itemId で一意、quantity を更新」を前提とする。
