# Spec: 認証（ユーザー登録・ログイン・セッション）

`docs/01_features.md` MVP「アカウント」に対応。

------------------------------------------------------------------------

## 0. 依存・横断

### 0.1 依存する spec

なし（認証は基盤 spec）。

### 0.2 提供する API / 利用者

| API | 用途 | 呼び出し元 |
|-----|------|-----------|
| register | 新規登録 | 登録画面 |
| login | ログイン | ログイン画面 |
| logout | ログアウト | ヘッダー・設定等 |
| getSession | セッション取得・検証 | 全保護画面（ミドルウェア／layout 経由） |

**横断前提**：他 spec で「保護画面」を定義する場合、`010_auth の getSession が有効であること` を前提条件に記載する。

## 1. 目的

- ユーザーがメールアドレスとパスワードで登録・ログインできること。
- ログイン状態をセッション（Cookie）で管理し、保護された画面へのアクセスを制御すること。
- ログアウトでセッションを破棄すること。

------------------------------------------------------------------------

## 2. 用語

- **User**：ユーザーアカウント（`docs/02_domain_model.md` 準拠）。email で一意。
- **セッション**：iron-session による暗号化 Cookie。ログイン時に userId を保持。
- **登録**：新規 User 作成。email 未登録であること、パスワード要件を満たすことが条件。
- **ログイン**：email + パスワードが正しい場合にセッションを確立。

------------------------------------------------------------------------

## 3. 入力（Input）

### 3.1 ユーザー登録

```json
{
  "email": "user@example.com",
  "password": "validPassword123",
  "name": "冒険者"
}
```

- `email`：必須。メール形式。255文字以内。既存ユーザーと重複不可。
- `password`：必須。8文字以上。MVPでは英数字のみ可（後で拡張）。
- `name`：任意。表示名。50文字以内。未指定時は null または空。

### 3.2 ログイン

```json
{
  "email": "user@example.com",
  "password": "validPassword123"
}
```

### 3.3 ログアウト

入力なし。セッション Cookie の有無のみが入力。

------------------------------------------------------------------------

## 4. 出力（Output）

### 4.1 登録・ログイン成功時

```json
{
  "success": true,
  "userId": "clx..."
}
```

- 登録成功時：User 作成後、自動でログイン（セッション確立）する。
- レスポンスとともに Set-Cookie でセッション Cookie を返す。

### 4.2 エラー時

```json
{
  "success": false,
  "error": "ERROR_CODE",
  "message": "ユーザー向けメッセージ"
}
```

- `ERROR_CODE` 例：`EMAIL_ALREADY_EXISTS`, `INVALID_CREDENTIALS`, `VALIDATION_ERROR`

### 4.3 ログアウト成功時

```json
{
  "success": true
}
```

- セッション Cookie を破棄する（空または期限切れで上書き）。

------------------------------------------------------------------------

## 5. ルール

### 5.1 登録

- email が既に存在する場合は `EMAIL_ALREADY_EXISTS` を返す。
- パスワードは保存前にハッシュ化する（bcrypt 等）。平文は保存しない。
- name が空文字の場合は null として保存する。

### 5.2 ログイン

- email に該当する User が存在し、パスワードが一致する場合のみ成功。
- 該当なし・パスワード不一致の場合は区別せず `INVALID_CREDENTIALS` を返す（アカウント存在のヒントを与えない）。

### 5.3 セッション

- iron-session を使用。Cookie 名は `remaria-session`（`lib/auth/session.ts` に定義）。
- セッションには `userId` のみ保持。有効期限は 7 日（延長は Next で検討）。
- 保護画面：セッションが無効または userId が存在しない場合はログイン画面へリダイレクト。

### 5.4 バリデーション

- email：RFC 5322 の簡易チェック（`/^[^\s@]+@[^\s@]+\.[^\s@]+$/` 相当）、空不可、最大 255 文字。
- password：8 文字以上、最大 72 文字（bcrypt 制限）。MVP では英数字のみ許可しても可。

------------------------------------------------------------------------

## 6. 処理フロー（概要）

### 6.1 登録

1. 入力検証（email 形式、password 長、name 長）
2. email 重複チェック
3. パスワードハッシュ化
4. User 作成（DB）
5. セッションに userId を設定
6. 成功レスポンス + Cookie 返却

### 6.2 ログイン

1. 入力検証
2. User 取得（email）
3. パスワード照合
4. 一致すればセッションに userId を設定
5. 成功レスポンス + Cookie 返却

### 6.3 ログアウト

1. セッションを破棄（`session.destroy()`）
2. 成功レスポンス

------------------------------------------------------------------------

## 7. 永続化データ / 一時データ

### 7.1 永続化するデータ

- **User**（DB）
  - id, email, passwordHash, name, createdAt, updatedAt
- **セッション**（Cookie 内の暗号化データ）
  - userId（復号化して取得）

### 7.2 保存しないデータ

- パスワード平文
- ログイン試行回数・ロック状態（MVP では不要、Later で検討）

`docs/04_persistence_principles.md` に従う。

------------------------------------------------------------------------

## 8. 例（最低3ケース）

### ケース1：登録・ログイン正常系

**登録入力：**

```json
{
  "email": "new@example.com",
  "password": "password123",
  "name": "テスト冒険者"
}
```

期待：User 作成、セッション確立、`success: true`、Cookie 付与。

**ログイン入力：**

```json
{
  "email": "new@example.com",
  "password": "password123"
}
```

期待：`success: true`、Cookie 付与。

### ケース2：email 重複

**登録入力：** 既存の email で登録

期待：`success: false`, `error: "EMAIL_ALREADY_EXISTS"`。User は作成されない。

### ケース3：ログイン認証失敗

**ログイン入力：**

```json
{
  "email": "unknown@example.com",
  "password": "wrongpassword"
}
```

期待：`success: false`, `error: "INVALID_CREDENTIALS"`。セッションは設定されない。

------------------------------------------------------------------------

## 9. エラー条件 / NG

| 条件 | 挙動 |
|------|------|
| 登録時 email 重複 | `EMAIL_ALREADY_EXISTS` |
| ログイン時 該当なし / パスワード不一致 | `INVALID_CREDENTIALS` |
| バリデーション違反（形式・長さ） | `VALIDATION_ERROR`、該当フィールドを message に含める |
| 未ログインで保護画面アクセス | ログイン画面へリダイレクト |
| 既ログイン状態でログイン/登録画面アクセス | ダッシュボード等へリダイレクト（任意、MVP では省略可） |

------------------------------------------------------------------------

## 10. テスト観点

- 登録：新規 email で User が作成され、パスワードがハッシュ化されて保存されること
- ログイン：正しい email + パスワードでセッションが確立されること
- ログイン：誤った認証情報で `INVALID_CREDENTIALS` が返ること
- 登録：重複 email で `EMAIL_ALREADY_EXISTS` が返ること
- ログアウト：セッションが破棄され、保護画面にアクセスできないこと
- バリデーション：不正な email / 短い password でエラーになること

------------------------------------------------------------------------

## 11. 画面仕様

`docs/07_ui_guidelines.md` のカラーパレット・Tailwind クラスに従う。
一般的な認証画面の構成とする（特別な演出は不要）。

------------------------------------------------------------------------

### 11.1 共通レイアウト（登録・ログイン画面）

| 項目 | 内容 |
|------|------|
| **レイアウト** | 画面中央にフォームを配置。縦方向に余白を確保 |
| **コンテナ** | カード風パネル（`bg-base-elevated`, `border-base-border`）でフォームを囲む |
| **幅** | モバイル〜デスクトップまで対応。最大幅 400px 程度が目安 |
| **フォーム** | 縦並び。項目間の余白を適度に |

------------------------------------------------------------------------

### 11.2 登録画面

| 項目 | 内容 |
|------|------|
| **URL** | `/register` |
| **前提** | 未ログイン（既ログイン時はダッシュボードへリダイレクト可） |
| **見出し** | 「アカウント登録」等 |
| **要素（上から順）** | email input（必須）, password input（必須）, name input（任意）, 登録ボタン, 「すでにアカウントをお持ちの方はログイン」リンク |
| **入力タイプ** | email: `type="email"`, password: `type="password"` |
| **呼び出す API** | register |
| **成功時** | `/dashboard`（または `/`）へリダイレクト |
| **エラー時** | フォーム直上または直下にエラーメッセージ表示。フォームは維持 |

------------------------------------------------------------------------

### 11.3 ログイン画面

| 項目 | 内容 |
|------|------|
| **URL** | `/login` |
| **前提** | 未ログイン（既ログイン時はダッシュボードへリダイレクト可） |
| **見出し** | 「ログイン」等 |
| **要素（上から順）** | email input（必須）, password input（必須）, ログインボタン, 「アカウントをお持ちでない方は登録」リンク |
| **入力タイプ** | email: `type="email"`, password: `type="password"` |
| **呼び出す API** | login |
| **成功時** | `/dashboard`（または `/`）へリダイレクト |
| **エラー時** | フォーム直上または直下にエラーメッセージ表示。フォームは維持 |

------------------------------------------------------------------------

### 11.4 ログアウト

| 項目 | 内容 |
|------|------|
| **配置** | ヘッダー・ナビゲーション等（保護画面に表示） |
| **要素** | ログアウトボタンまたはリンク |
| **呼び出す API** | logout |
| **成功時** | `/login` へリダイレクト |

------------------------------------------------------------------------

### 11.5 保護画面でのリダイレクト

| 項目 | 内容 |
|------|------|
| **対象** | ログイン必須の全画面 |
| **判定** | getSession で userId が取得できない場合 |
| **挙動** | `/login` へリダイレクト |

------------------------------------------------------------------------

### 11.6 導線（一般的な遷移）

```
未ログイン状態:
  トップ(/)
    ├→ /login（ログイン画面へのリンク）
    └→ /register（登録画面へのリンク）

/login と /register の間は相互リンクで行き来可能

ログイン後:
  任意の保護画面 → ログアウト → /login
```

## 付録：スキーマ変更

認証に必要な Prisma スキーマ変更：

```prisma
model User {
  id           String   @id @default(cuid())
  email        String   @unique
  passwordHash String   // 追加
  name         String?
  createdAt    DateTime @default(now())
  updatedAt    DateTime @updatedAt

  playerCharacter PlayerCharacter?
}
```
