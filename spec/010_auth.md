# Spec: 認証（ユーザー登録・ログイン・セッション）

`docs/01_features.md` MVP「アカウント」に対応。

------------------------------------------------------------------------

## 0. 依存・横断

### 0.1 依存する spec

なし（認証は基盤 spec）。

### 0.2 提供する API / 利用者

| API | 用途 | 呼び出し元 |
|-----|------|-----------|
| register | 新規登録 | 登録画面 |
| login | ログイン | ログイン画面 |

## 1. 目的

- ユーザーがメールアドレスとパスワードで登録・ログインできること。
- ログイン状態をセッション（Cookie）で管理し、保護された画面へのアクセスを制御すること。
- ログアウトでセッションを破棄すること。

------------------------------------------------------------------------

## 2. 用語

- **User**：ユーザーアカウント（`docs/02_domain_model.md` 準拠）。email で一意。任意ID（accountId）も一意（指定時のみ）。
- **セッション**：iron-session による暗号化 Cookie。ログイン時に userId を保持。
- **登録**：新規 User 作成。メールアドレス・ID（英数字・必須・重複不可）・名前（必須・重複可）で登録。email 未登録であること、accountId 未使用であること、パスワード要件を満たすことが条件。
- **ログイン**：email + パスワードが正しい場合にセッションを確立。

------------------------------------------------------------------------

## 3. 入力（Input）

### 3.1 ユーザー登録

```json
{
  "email": "user@example.com",
  "accountId": "my_id_123",
  "password": "validPassword123",
  "name": "冒険者"
}
```

- `email`：必須。メール形式。255文字以内。既存ユーザーと**重複不可**。
- `accountId`：必須。英数字のみ。既存ユーザーと**重複不可**。フォーマットは英数字（例: 3〜32文字など実装で規定）。
- `password`：必須。8文字以上。MVPでは英数字のみ可（後で拡張）。
- `name`：必須。表示名。**おおよそ全角 12 文字・半角 24 文字（UTF-8 約 24 バイト）以内**。**重複可**。この値が**主人公の表示名**としても使われる（User にのみ保持、二重管理しない）。

### 3.2 ログイン

```json
{
  "email": "user@example.com",
  "password": "validPassword123"
}
```

### 3.3 ログアウト

入力なし。セッション Cookie の有無のみが入力。

------------------------------------------------------------------------

## 4. 出力（Output）

### 4.1 登録・ログイン成功時

```json
{
  "success": true,
  "userId": "clx..."
}
```

- 登録成功時：User 作成後、自動でログイン（セッション確立）する。
- レスポンスとともに Set-Cookie でセッション Cookie を返す。

### 4.2 エラー時

```json
{
  "success": false,
  "error": "ERROR_CODE",
  "message": "ユーザー向けメッセージ"
}
```

- `ERROR_CODE` 例：`EMAIL_ALREADY_EXISTS`, `ACCOUNT_ID_ALREADY_EXISTS`, `INVALID_CREDENTIALS`, `VALIDATION_ERROR`

### 4.3 ログアウト成功時

```json
{
  "success": true
}
```

- セッション Cookie を破棄する（空または期限切れで上書き）。

------------------------------------------------------------------------

## 5. ルール

### 5.1 登録

- email が既に存在する場合は `EMAIL_ALREADY_EXISTS` を返す。
- accountId が既に存在する場合は `ACCOUNT_ID_ALREADY_EXISTS` を返す。
- パスワードは保存前にハッシュ化する（bcrypt 等）。平文は保存しない。
- name は必須。空文字は不可（名前は重複可のため一意チェックは行わない）。この name が主人公の表示名としても使われる（User にのみ保持）。

### 5.2 ログイン

- email に該当する User が存在し、パスワードが一致する場合のみ成功。
- 該当なし・パスワード不一致の場合は区別せず `INVALID_CREDENTIALS` を返す（アカウント存在のヒントを与えない）。

### 5.3 セッション

- iron-session を使用。Cookie 名は `remaria-session`（`lib/auth/session.ts` に定義）。
- セッションには `userId` のみ保持。有効期限は 7 日（延長は Next で検討）。
- 保護画面：セッションが無効または userId が存在しない場合はログイン画面へリダイレクト。

### 5.4 バリデーション

- email：RFC 5322 の簡易チェック（`/^[^\s@]+@[^\s@]+\.[^\s@]+$/` 相当）、空不可、最大 255 文字。
- accountId：必須。英数字のみ（例: `^[a-zA-Z0-9_]+$`）、長さは実装で規定（例: 3〜32 文字）。
- password：8 文字以上、最大 72 文字（bcrypt 制限）。MVP では英数字のみ許可しても可。

------------------------------------------------------------------------

## 6. 処理フロー（概要）

### 6.1 登録

1. 入力検証（email 形式、accountId 形式・長さ、password 長、name 長）
2. email 重複チェック → 重複時は `EMAIL_ALREADY_EXISTS`
3. accountId 重複チェック → 重複時は `ACCOUNT_ID_ALREADY_EXISTS`
4. パスワードハッシュ化
5. User 作成（DB）
6. セッションに userId を設定
7. 成功レスポンス + Cookie 返却

### 6.2 ログイン

1. 入力検証
2. User 取得（email）
3. パスワード照合
4. 一致すればセッションに userId を設定
5. 成功レスポンス + Cookie 返却

### 6.3 ログアウト

1. セッションを破棄（`session.destroy()`）
2. 成功レスポンス

------------------------------------------------------------------------

## 7. 永続化データ / 一時データ

### 7.1 永続化するデータ

- **User**（DB）
  - id, email, accountId（必須・一意）, passwordHash, name（必須。主人公の表示名もこれを使用）, createdAt, updatedAt
  - **lastActiveAt**（DateTime, NULL可）：直近で「操作があった」とみなした日時。プレイ中人数表示・管理画面で参照。更新はスロットリング（同一ユーザーは最低 1 分間隔）で行う。
- **セッション**（Cookie 内の暗号化データ）
  - userId（復号化して取得）

### 7.2 保存しないデータ

- パスワード平文
- ログイン試行回数・ロック状態（MVP では不要、Later で検討）

`docs/04_persistence_principles.md` に従う。

### 7.3 管理用アカウント

- 管理画面（`/dashboard/admin/*`）に入室できるのは、**環境変数 `ADMIN_EMAIL` で指定されたメールアドレスに一致する User 1 件のみ**。未設定時は `test1@example.com`。
- 該当 User はシードで「管理人」として作成される。パスワードは環境変数 `ADMIN_PASSWORD` またはシード時のランダム生成。詳細は **`manage/ADMIN_ACCOUNT.md`**。

## 8. 例（最低3ケース）

### ケース1：登録・ログイン正常系

**登録入力：**

```json
{
  "email": "new@example.com",
  "accountId": "test_user_01",
  "password": "password123",
  "name": "テスト冒険者"
}
```

期待：User 作成、セッション確立、`success: true`、Cookie 付与。

**ログイン入力：**

```json
{
  "email": "new@example.com",
  "password": "password123"
}
```

期待：`success: true`、Cookie 付与。

### ケース2：email 重複

**登録入力：** 既存の email で登録

期待：`success: false`, `error: "EMAIL_ALREADY_EXISTS"`。User は作成されない。

### ケース2b：accountId 重複

**登録入力：** accountId に既存ユーザーが使用中の値を指定して登録

期待：`success: false`, `error: "ACCOUNT_ID_ALREADY_EXISTS"`。User は作成されない。

### ケース3：ログイン認証失敗

**ログイン入力：**

```json
{
  "email": "unknown@example.com",
  "password": "wrongpassword"
}
```

期待：`success: false`, `error: "INVALID_CREDENTIALS"`。セッションは設定されない。

## 9. エラー条件 / NG

| 条件 | 挙動 |
|------|------|
| 登録時 email 重複 | `EMAIL_ALREADY_EXISTS` |
| 登録時 accountId 重複 | `ACCOUNT_ID_ALREADY_EXISTS` |
| ログイン時 該当なし / パスワード不一致 | `INVALID_CREDENTIALS` |
| バリデーション違反（形式・長さ） | `VALIDATION_ERROR`、該当フィールドを message に含める |
| 未ログインで保護画面アクセス | ログイン画面へリダイレクト |
| 既ログイン状態でログイン/登録画面アクセス | ダッシュボード等へリダイレクト（任意、MVP では省略可） |

------------------------------------------------------------------------

## 10. テスト観点

- 登録：新規 email（および任意の accountId・name）で User が作成され、パスワードがハッシュ化されて保存されること
- ログイン：正しい email + パスワードでセッションが確立されること
- ログイン：誤った認証情報で `INVALID_CREDENTIALS` が返ること
- 登録：重複 email で `EMAIL_ALREADY_EXISTS` が返ること
- 登録：重複 accountId で `ACCOUNT_ID_ALREADY_EXISTS` が返ること
- ログアウト：セッションが破棄され、保護画面にアクセスできないこと
- バリデーション：不正な email / 短い password でエラーになること

## 11. 画面仕様

`docs/07_ui_guidelines.md` のカラーパレット・Tailwind クラスに従う。
一般的な認証画面の構成とする（特別な演出は不要）。

------------------------------------------------------------------------

### 11.1 共通レイアウト（登録・ログイン画面）

| 項目 | 内容 |
|------|------|
| **レイアウト** | 画面中央にフォームを配置。縦方向に余白を確保 |
| **コンテナ** | カード風パネル（`bg-base-elevated`, `border-base-border`）でフォームを囲む |
| **幅** | モバイル〜デスクトップまで対応。最大幅 400px 程度が目安 |
| **フォーム** | 縦並び。項目間の余白を適度に |

------------------------------------------------------------------------

### 11.2 登録画面

| 項目 | 内容 |
|------|------|
| **URL** | `/register` |
| **前提** | 未ログイン（既ログイン時はダッシュボードへリダイレクト可） |
| **見出し** | 「アカウント登録」等 |
| **要素（上から順）** | email input（必須）, accountId input（必須・英数字）, password input（必須）, name input（必須）, 登録ボタン, 「すでにアカウントをお持ちの方はログイン」リンク |
| **入力タイプ** | email: `type="email"`, password: `type="password"` |
| **呼び出す API** | register |
| **成功時** | `/dashboard`（または `/`）へリダイレクト |
| **エラー時** | フォーム直上または直下にエラーメッセージ表示。フォームは維持 |

------------------------------------------------------------------------

### 11.3 ログイン画面

| 項目 | 内容 |
|------|------|
| **URL** | `/login` |
| **前提** | 未ログイン（既ログイン時はダッシュボードへリダイレクト可） |
| **ウェルカムキャラクター** | `public/icons/` 内のアイコンのいずれかを**ランダムに1つ**表示する。配置はフォームの上。参照パスは `/icons/{ファイル名}`（例: `/icons/2.gif`）。表示対象の一覧（例: 1.gif～5.gif）は実装で定数管理する |
| **見出し** | 「ログイン」等 |
| **要素（上から順）** | ウェルカムキャラクター（ランダム1点）, 見出し, email input（必須）, password input（必須）, ログインボタン, 「アカウントをお持ちでない方は登録」リンク |
| **入力タイプ** | email: `type="email"`, password: `type="password"` |
| **呼び出す API** | login |
| **成功時** | `/dashboard`（または `/`）へリダイレクト |
| **エラー時** | フォーム直上または直下にエラーメッセージ表示。フォームは維持 |

#### 11.3.1 ウェルカムキャラクター（詳細）

- **アセット配置**: `public/icons/` の直下に GIF を配置する。
- **表示対象一覧**: 実装では定数で管理する。候補は `1.gif`, `2.gif`, `3.gif`, `4.gif`, `5.gif` とする（追加・削除する場合はこの一覧を実装と本 spec で揃える）。
- **表示ルール**: ログイン画面表示ごとに、上記一覧から**ランダムに1つ**選んで表示する。同一表示中は同じ画像を維持する（リロード・再訪問時に別の画像になり得る）。
- **参照パス**: `/icons/{ファイル名}`（例: `/icons/2.gif`）。
- **配置**: フォームの上、中央寄せ。サイズはレイアウトに合わせて調整（例: 高さ上限 8rem）。

------------------------------------------------------------------------

### 11.4 ログアウト

| 項目 | 内容 |
|------|------|
| **配置** | ヘッダー・ナビゲーション等（保護画面に表示） |
| **要素** | ログアウトボタンまたはリンク |
| **呼び出す API** | logout |
| **成功時** | `/login` へリダイレクト |

------------------------------------------------------------------------

### 11.5 保護画面でのリダイレクト

| 項目 | 内容 |
|------|------|
| **対象** | ログイン必須の全画面 |
| **判定** | getSession で userId が取得できない場合 |
| **挙動** | `/login` へリダイレクト |

------------------------------------------------------------------------

### 11.6 アクティブユーザー数・最終アクティブ（lastActiveAt）

| 項目 | 内容 |
|------|------|
| **目的** | 「いまゲームで遊んでいる人数」を直近5分以内に操作があったユーザー数で表示する。管理画面では各ユーザーの最終アクティブ日時を参照可能にする。 |
| **User.lastActiveAt** | 保護パス（/dashboard, /character, /battle）へのアクセスなど「操作」があったときに更新する。同一ユーザーは **1 分に 1 回まで** 更新（スロットリング）して DB 負荷を抑える。 |
| **更新タイミング** | ルート layout で getSession により userId が取れた場合に `touchUserActivity(userId)` を呼ぶ。 |
| **人数取得** | `getActiveUserCountLast5Min()`：`lastActiveAt >= NOW() - INTERVAL 5 MINUTE` のユーザー数を返す。 |
| **表示** | ヘッダーに「いま〇人がプレイ中」等を表示（未ログインでも表示してよい）。管理画面のユーザ一覧に「最終アクティブ」列を追加し lastActiveAt を表示する。 |

------------------------------------------------------------------------

### 11.7 導線（一般的な遷移）

```
未ログイン状態:
  トップ(/)
    ├→ /login（ログイン画面へのリンク）
    └→ /register（登録画面へのリンク）

/login と /register の間は相互リンクで行き来可能

ログイン後:
  任意の保護画面 → ログアウト → /login
```
