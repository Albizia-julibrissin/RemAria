# Spec: 作戦スロット（キャラごと）と編集画面

`docs/14_battle_attributes_tactics.md`・`docs/14_battle_tactics_impl_draft.md` に基づき、作戦スロット（主語−条件−行動×10）を **キャラごとに永続**し、編集画面で **3 人分を同時に表示・編集**できるようにする。

------------------------------------------------------------------------

## 0. 作戦室（画面・要件）

- **作戦室**とは、戦闘に移る前に **作戦スロットを定義・変更するための画面（または機能単位）** を指す。
- **ユーザーは作戦を変えられる必要がある**。つまり、作戦室から各キャラのスロット（主語・条件・行動）を閲覧・編集・保存できることが必須要件である。
- 利用タイミング：戦闘開始前の編成・準備の一環。編成で選んだ 3 キャラの作戦を、いつでも（戦闘前だけでなく必要に応じて）編集可能。

### 0.1 画面フロー（イメージ）

1. **作戦室を開く** → まず **呼び出すパーティプリセットを選択**する。
2. **プリセットを選ぶ**と、
   - **主人公枠は固定で設定済み**（強制出撃）。残り 2 枠で **仲間** と **メカ** を選ぶ。
   - 3 枠が決まった時点で、その 3 キャラ分の作戦スロットが編集対象になる。
3. **各枠（キャラ）ごとに作戦設定**：主語・条件・行動を **プルダウン** でセットする（orderIndex 1～10）。
4. **全員設定して保存** → **パーティプリセット 1 つとして保存**する。保存単位は **パーティ単位のプリセット**（編成 3 人＋各キャラの作戦 10 件×3 を一括で反映）。

※ **プリセットごとに別作戦**を実現する拡張は **spec/063_tactics_per_preset.md** で定義する。063 採用後は作戦は **PresetTacticSlot**（partyPresetId + characterId + orderIndex）で永続し、取得・保存・戦闘参照はすべて PresetTacticSlot のみを使用する。現状（063 未適用）では TacticSlot（キャラ単位）で永続しており、同一キャラは全プリセットで作戦が共通。

### 0.2 操作詳細

- **キャラクター選択済みの枠**にだけ、作戦設定 UI（主語・条件・行動のプルダウン）を表示する。
- 未選択枠（仲間・メカが未選択など）は作戦編集不可、または枠を選んでから編集。

### 0.4 編成変更（仲間・メカの差し替え）時のデータ取得

- 画面上で仲間枠・メカ枠のドロップダウンを別キャラに変更した場合、**その時点の編成 3 人**のうち、まだ作戦・習得スキルのデータを持っていないキャラがいることがある。
- その場合、クライアントは **不足しているキャラ分だけ** `getTacticsForCharacters` と `getBattleSkillsForCharacters` を呼び、取得結果をクライアントの state にマージする。これにより、ページ再読み込みなしで切り替え先の作戦・スキルを表示・編集できる。
- スキル一覧（編成メンバーのスキルカタログ）も、編成変更後に **現在の 3 人** で `getTacticsSkillCatalogForCharacters` を呼び直して更新してよい。

### 0.3 行動（スキル）選択の UX（スキル増加時）

主語・条件・行動はいずれもプルダウン想定。**スキルが増えると行動のプルダウンが長くなり選びづらくなる**ため、以下を検討する。

| 案 | 内容 | 備考 |
|----|------|------|
| **検索付きコンボボックス** | 入力で絞り込み。スキル名・読みの一部で検索。 | 実装コストは中。多くのスキルでも使いやすい。 |
| **種別でフィルタ** | 先に「物理 / 魔法 / 補助」で絞ってからスキル一覧を表示。 | 30 種程度なら 10 個ずつに分割されて見やすい。 |
| **グループ表示** | プルダウン内で「物理攻撃」「魔法攻撃」「回復・バフ」などグループ見出しで分割。 | 既存の category / battleSkillType を利用可能。 |
| **よく使う / お気に入り** | 直近使用や「お気に入り」を上部に固定。 | 運用でスキルが増えた後の拡張向け。 |
| **2 段階選択** | 1 段階目で「通常攻撃 / スキル」を選び、スキルなら 2 段階目で種別→一覧。 | プルダウンを短く保てる。 |

**推奨（初期実装）**：**種別フィルタ（物理・魔法・補助）＋ グループ表示** で一覧を短くし、必要なら **検索付きコンボボックス** を追加する。スキル数が 30 程度なら種別＋グループで十分な場合が多い。

## 1. 依存・横断

### 1.1 依存する spec

- **010_auth**：ログイン済み前提。
- **025_character_list**：ユーザーが所持する Character を取得できる。
- **015_protagonist_creation**：主人公が存在する。

### 1.2 提供する API / 利用者

| API | 用途 | 呼び出し元 | 063 採用後 |
|-----|------|-----------|------------|
| getTacticsForCharacters / getTacticsForPreset | 指定した 3 キャラ分（またはプリセットに紐づく）作戦スロットを取得 | 作戦室（プリセット選択後）、編成変更時追加取得 | getTacticsForPreset(presetId) で PresetTacticSlot から取得 |
| upsertTacticsForCharacter / savePresetWithTactics | 1 キャラ分の一括保存、またはプリセット＋3 人分の一括保存 | 作戦室（保存時） | savePresetWithTactics 内で PresetTacticSlot のみ一括置換（063） |
| （プリセット取得・更新） | プリセット一覧の取得、選択中プリセットの編成（3 枠）更新 | 作戦室（別 spec / 既存のパーティ編成 API に依存） | 変更なし |

作戦室での「保存」は **パーティプリセット単位**。063 採用後は編成・名前・列の更新に加え、**そのプリセットに紐づく PresetTacticSlot を一括置換**する（spec/063）。

### 1.3 敵側の作戦スロット（横断）

**同じ主語・条件・行動の仕様**が **敵マスタ（Enemy）** 側でも使われる。spec/050 で EnemyTacticSlot として敵種ごとに最大 10 件を持ち、管理画面「敵マスタ編集」で編集する。選択肢の定数（`tactics-constants.ts`）と評価ロジック（spec/040）は味方・敵で共有。**作戦スロット関連の仕様変更・選択肢追加時は、本 spec（作戦室）と spec/050（敵マスタ編集）の両方に影響するため、両方を確認すること。** まとめは `docs/14_tactics_slot_shared.md` を参照。

## 2. 目的

- 各キャラが「自分の作戦スロット 10 個」を持ち、戦闘時の行動選択（オート）に利用できるようにする。
- 編集UIは **3 人を選び、3 人分の作戦を 1 画面で確認・編集**できる（パーティ全体のコンボを見渡せる）。

------------------------------------------------------------------------

## 3. 用語

- **作戦スロット**：orderIndex 1～10 の優先順位リスト。主語・条件・行動からなる。
- **主語（subject）**：self / any_ally / any_enemy / front_enemy（将来拡張あり）。
- **条件種別（conditionKind）**：hpBelowPercent 等。param は JSON で持つ。
- **行動（actionType）**：normal_attack または skill + skillId。

------------------------------------------------------------------------

## 4. 入力（Input）

### 4.1 getTacticsForCharacters

```json
{
  "characterIds": ["cuidA", "cuidB", "cuidC"]
}
```

### 4.2 upsertTacticsForCharacter

```json
{
  "characterId": "cuidA",
  "slots": [
    {
      "orderIndex": 1,
      "subject": "any_enemy",
      "conditionKind": "subject_has_attr_state",
      "conditionParam": { "attr": "crush" },
      "actionType": "skill",
      "skillId": "cuidSkillHighSpear"
    }
  ]
}
```

------------------------------------------------------------------------

## 5. 出力（Output）

### 5.1 getTacticsForCharacters

```json
{
  "tactics": [
    {
      "characterId": "cuidA",
      "slots": [ /* orderIndex 昇順 最大10件 */ ]
    }
  ]
}
```

## 6. ルール

### 6.1 永続化

- **063 未適用時**：テーブル TacticSlot（characterId, orderIndex, subject, conditionKind, conditionParam, actionType, skillId）。@@unique([characterId, orderIndex])。characterId は userId の所有物。
- **063 適用後**：永続先は **PresetTacticSlot**（spec/063）。TacticSlot は廃止。取得・保存・戦闘参照はすべて PresetTacticSlot を使用する。

### 6.2 デフォルト

- 作戦が未設定（0件）のキャラは「全スロット無し」とみなす。
- 戦闘時：条件に一致するスロットが無い場合は通常攻撃。

### 6.3 スキル選択時の不発（参照）

- スキルが選ばれても MP 不足なら不発（spec/038 の 5.1）。
- 不発時は次スロットにフォールバックしない（そのターンは不発で行動済み）。

### 6.4 条件とターゲット抽選の関係

- 条件で見る相手（subject）と、実際に攻撃が当たる相手は別。ターゲットは列ウェイトで抽選（`docs/14_battle_attributes_tactics.md` 6.6）。

------------------------------------------------------------------------

## 6.5 戦闘時の行動決定（参照）

- 味方のターンが来たとき、**そのキャラの作戦スロットを orderIndex 昇順で評価**し、最初に条件を満たしたスロットの行動を採用する。
- **条件の評価方法・主語の解決・コンテキスト**は **spec/040_tactic_slot_evaluation.md** に定義する。実装時は 040 を参照すること。

------------------------------------------------------------------------

## 7. 処理フロー（概要）

### 7.1 取得

1. セッションで userId を取得。
2. characterIds が user の所有物か検証。
3. 各 characterId の slots を orderIndex 昇順で返す。

### 7.2 保存

1. セッションで userId を取得。
2. characterId が user の所有物か検証。
3. slots のバリデーション（orderIndex 1～10、subject/kind が既知、param の整合、skillId の存在など）。
4. その characterId の TacticSlot を **一括置換**（deleteMany + createMany 等）。

------------------------------------------------------------------------

## 8. 永続化データ / 一時データ

### 8.1 永続化するデータ

- TacticSlot（上記）

### 8.2 保存しないデータ

- 戦闘中の評価結果、条件評価の中間状態。

------------------------------------------------------------------------

## 9. 例（最低3ケース）

### ケース1：ブラックホール→ハイスピア

- キャラA：any_enemy が crush 状態ならハイスピア
- キャラB：MP>=20% ならブラックホール
- 期待：圧縮が付いた敵が存在する状況でAがハイスピアを選ぶ。ただしターゲットは抽選なので圧縮持ちに当たる保証はない。

### ケース2：未設定

- slots が空
- 期待：戦闘時は常に通常攻撃（条件一致なし）

### ケース3：編集UI

- 画面で3キャラを選択し、3人分のスロットを並列表示して編集できる
