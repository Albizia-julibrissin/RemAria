# 運営ログの整理・導入タイミング

**目的**: 運営で「あったほうがよいログ」を一覧化し、**いつ導入するか**を優先度・コストと合わせて整理する。  
既存の [SECURITY_READINESS.md](./SECURITY_READINESS.md) や [PRODUCTION_RELEASE_GUIDE.md](./PRODUCTION_RELEASE_GUIDE.md) と重複する部分はそちらを正本とし、本ドキュメントでは「運営視点のログ一覧」と「導入タイミング」に絞る。

### 1.1 プレイヤー行動記録（探索回数など）

「探索回数」「どのエリアが何回プレイされたか」などは、**いまあるデータを集計する**だけでかなりカバーできる。

**探索終了時の Expedition の扱い（実装確認済み）**  
終了時（帰還 or 中断）に Expedition は**削除していない**。`finishExploration` は `state: "finished"` に更新、`abortCurrentExpedition` は `state: "aborted"` に更新するだけ。そのため **Expedition の件数で探索回数を集計できる**。将来「終了済みは削除する」方針に変える場合は、探索回数用に別テーブル（例: ユーザー×エリアのカウント or 1 回 1 行のログ）を用意する必要がある。

| 知りたいこと | 現状でできること | 備考 |
|--------------|------------------|------|
| 探索回数（ユーザー単位） | `Expedition` を userId で COUNT（state は問わず、または finished のみなど） | 1 回の出撃 = 1 Expedition。 |
| 探索回数（エリア単位） | `Expedition` を areaId で COUNT | 人気エリア・バランス調整に使える。 |
| いつ頃プレイされたか | `Expedition.createdAt` で日付集計 | 日別のアクティブ数など。 |
| 戦闘回数 | 戦闘結果を保持しているテーブルがあれば同様に COUNT | 練習戦闘は別テーブル/ログの有無は要確認。 |

**導入の考え方**

- **MVP 〜 Next**: 専用テーブルは作らず、**管理画面に「集計クエリ」や簡易ビューを 1 つ用意する**程度でよい（例: ユーザー別・エリア別の探索回数、直近 N 日分）。既存の Expedition 等をそのまま集計する。
- **LATER**: バランス調整や分析を本格化するタイミングで、必要なら「行動イベントを 1 行ずつ書くログテーブル」を検討（毎探索・毎戦闘を 1 レコードにすると、時系列分析や不正検知がしやすくなるが、データ量は増える）。

**Expedition が増えすぎる場合（ヘビープレイヤー）**  
「開始のたびに 1 行」なので、1 ユーザーが 1 日 1000 回プレイすると Expedition が 1000 行/日/人だけ増える。ユーザー数×日数が増えるとテーブルが肥大化し、集計やインデックスが重くなりやすい。

- **方針候補（LATER で検討）**
  - **集計用カウンタを別テーブルに持つ**: 例として `UserExplorationCount`（userId, areaId, totalCount や 日別集計）を用意し、**探索終了時（finishExploration）に +1 する**。探索回数はこのテーブルを見る。
  - **終了済み Expedition は残さない**: 終了済み（state = finished / aborted）は一定期間後に**削除 or 別ストレージにアーカイブ**する。進行中・直近 N 件だけ Expedition に残す。回数は上記カウンタに依存。
- いつやるか: 件数が実際に問題になってから（例: Expedition が数百万行超えてクエリが遅い、バックアップが重い）でよい。その時点で「終了時にカウンタ更新」と「古い Expedition の削除/アーカイブ」を同時に入れると、行増やしスタイルのままでも重くならない。

**1 行使い回し vs 削除＋新規作成（検討用）**  
「行を増やさない」ために、**finish した行をきれいにリセットして次の start で再利用する**案と、**finish 時に削除して start で毎回 CREATE** する案の比較。

| 観点 | 1 行使い回し（UPDATE でリセット） | 削除＋新規作成（DELETE + CREATE） |
|------|-----------------------------------|-----------------------------------|
| **行数** | 1 ユーザーあたり実質 1 行（または進行中のみ 1 行）。増えない。 | 進行中＋「直近だけ残す」にすれば行数は有界。終了即削除なら進行中分だけ。 |
| **探索回数** | Expedition の件数では取れない。**別カウンタ必須**（User のカラム or UserExplorationCount 等）。 | 終了即削除なら同様に**別カウンタ必須**。直近 N 件を残す運用なら「直近分は COUNT 可能」だが総数はカウンタ。 |
| **履歴** | 直前 1 回の内容しか残らない。サポートで「先週の探索結果」を見たい場合は別ログが必要。 | 終了即削除なら同じ。直近 N 件残すなら、その範囲だけ履歴が残る。 |
| **実装** | start: 自分の `state in (finished, aborted)` な行があればそれを UPDATE で初期状態に上書き、なければ CREATE。finish: 従来どおり UPDATE state=finished。 | start: 従来どおり CREATE。finish: 報酬等を反映したあと **DELETE**（または state だけ更新してバッチで後から削除）。 |
| **同時進行** | 1 ユーザー 1 行前提。将来「複数エリアを並行」にするなら行数＝並行数になり、使い回しの利点は薄れる。 | 1 run = 1 行のまま。並行対応時も「進行中が複数行」で自然。 |
| **まとめ** | 行数は最小。代わりに回数・履歴はすべて別設計（カウンタ・ログ）に寄せる。 | 1 run = 1 行の意味が保たれ、直近だけ残す運用と相性がよい。回数はやはりカウンタが必要。 |

**結論の目安**: どちらを選んでも「探索回数」は**別カウンタ**が必要になる。**使い回し**はテーブルサイズを最も抑えられるが、履歴が残らないので運用・サポートで「直近 1 回以外」を見るなら別の仕組みが要る。**削除＋新規**は「1 回 = 1 行」のモデルを保ったまま、終了後に削除 or 一定期間で消すことで増えすぎを防げる。どちらがよいかは「直近 N 回の run を DB に残したいか」「行数より履歴の手軽さを優先するか」で決めるとよい。

**方針（戦闘内容ログを取らない前提）**: 戦闘の内容ログをいまは取らない前提なら、**カウンタを別に用意して UPDATE で 1 行使い回す**方針でよい。Expedition は「進行中の 1 回分」だけ持ち、終了時にカウンタ（例: User の explorationCount や UserExplorationCount テーブル）を +1 し、同じ行をリセットして次の start で再利用する。レスポンスも行数も抑えられ、履歴が不要ならこれで十分。

## 2. 導入タイミングの目安

### 2.1 MVP 前〜MVP 直後（今のフェーズ）

| ログ | やること | 理由 |
|------|----------|------|
| **通貨取引** | 既存の `CurrencyTransaction` で、**通貨が増減する処理をすべてここに通す**。reason / referenceType / referenceId を統一して記録。 | 課金前でも「何で増減したか」の追跡ができると、不具合調査・バランス調整に役立つ。 |
| **本番DB操作** | **手動でよい**。「〇月〇日 本番に migrate deploy」「〇月〇日 マスタ同期（〇〇を追加）」などを manage 配下のメモや RELEASE チェックリストの実行メモに残す。 | 自動化しなくても、誰がいつ何をしたかを残す習慣をつけておく。 |

**導入しないでよいもの（この段階）**: ログイン試行ログ、管理画面操作ログ、専用の「実行履歴テーブル」。コストをかけずに運用でカバーできる範囲にとどめる。

---

### 2.2 課金を入れる前（PRE-PAID）

| ログ | やること | 理由 |
|------|----------|------|
| **通貨取引の強化** | SECURITY_READINESS のとおり、**beforeBalance / afterBalance**（または delta と整合する残高）を記録。Order との紐づけ（有償付与時）。reason を **reasonCode** 化して一覧管理。 | 課金周りのトラブル時に対象ユーザーの履歴を追いやすくする。 |
| **ログイン試行** | 失敗回数カウント＋**運営用ログ**（失敗時刻・IP・email または userId）。レートリミットとセットで導入。 | 不正アクセス試行・ credential stuffing の検知。SECURITY_READINESS §1 に記載済み。 |
| **本番DB操作** | 引き続き手動メモで可。必要なら「リリースメモ」を 1 ファイル（例: `manage/release_notes.md`）に日付・内容だけ追記する運用で十分。 | テーブルやツールを用意するより、軽い運用で続ける。 |

---

### 2.3 有償課金開始後〜運営が複数人になりそうな頃

| ログ | やること | 理由 |
|------|----------|------|
| **通貨履歴の運営ビュー** | 特定ユーザーの `CurrencyTransaction` 一覧を管理画面や簡易コマンドで確認できるようにする。 | 問い合わせ対応・不正検知。SECURITY_READINESS §5（LATER）に対応。 |
| **管理画面操作ログ** | マスタ編集時に「誰（userId）・いつ・どのテーブル/id・何を変更したか」を 1 テーブルに書き足す。 | 誤編集の原因追及・監査。管理画面の利用者が増えてからでよい。 |

---

### 2.4 規模が出てきたら（LATER）

| ログ | やること | 理由 |
|------|----------|------|
| **エラー・監視** | Sentry / APM 等で 500・未処理例外・遅延を検知。 | 障害の早期発見。SECURITY_READINESS §4。 |
| **レートリミットとログ** | チャット・クラフト等のレートリミット実装と同時に、制限発動時のログ（userId, IP, 時刻, 種別）を残す。 | 濫用・荒らしの傾向把握。 |

## 3. まとめ（いつ何を入れるか）

| タイミング | 入れるもの | 入れない／後回し |
|------------|------------|-------------------|
| **MVP 前〜直後** | 通貨取引の「全経路で CurrencyTransaction 記録」の徹底、本番DB操作の手動メモ | ログイン試行、管理画面操作、自動実行履歴DB |
| **課金前（PRE-PAID）** | 通貨取引の before/after・Order 紐づけ、ログイン試行（失敗）ログ | 管理画面操作ログ、運営ビュー（課金開始後で可） |
| **課金開始後** | 通貨履歴の運営ビュー、必要なら管理画面操作ログ | — |
| **LATER** | エラー監視、レートリミット＋ログ、必要ならプレイヤー行動のイベントログ（1行動1行） | — |

**プレイヤー行動（探索回数など）**: 専用ログは作らず、**既存の Expedition などを集計**して「探索回数」「エリア別回数」を見られるようにする（管理画面の簡易ビュー or 集計クエリ）と十分。本格的な分析用のイベントログは LATER でよい。

「運営のログ」は**まずは通貨と認証まわり**を課金に合わせて整え、**DB操作・デプロイは手動メモ**で始め、**プレイヤー行動は既存データの集計**で見える化し、**管理画面の監査や監視は規模に応じて**追加する、という段階でよい。
