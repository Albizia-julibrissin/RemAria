# 管理画面「本番マスタ更新」機能の設計メモ

**目的**: マスタの本番反映を、CLI（`db:sync-masters`）に頼らず、管理画面から「選択したマスタだけ・厳重確認のうえ」で実行できるようにする。

**正式仕様**: `spec/095_admin_master_production_sync.md`

---

## 1. 現状

- マスタの正本はローカルで管理画面編集した DB。
- 本番への反映は **`npm run db:sync-masters`**（SOURCE=ローカル、TARGET=本番 URL）を Cursor に依頼して実行。
- スクリプトは `prisma/sync-masters-to-target.ts`。テーブル一覧は **MASTER_DELEGATES_IN_ORDER** で依存順に固定。スキーマ変更のたびにこの一覧の手入れが必要。

---

## 2. 提案（管理画面で本番更新）

- **管理画面に「本番マスタ更新」画面を追加**する。
- **確認を厳重にする**（複数ステップ・確認文言の入力など）。
- **更新したいマスタを選択**してから実行する（全テーブル一括ではなく、必要なものだけ流す）。

### 2.1 メリット

| 項目 | 内容 |
|------|------|
| 選択同期 | 変更したマスタだけ選んで流せる。影響範囲が明確になり、リスクを減らせる。 |
| 運用のしやすさ | Cursor に依頼せず、管理者がブラウザから実行できる（実行環境の条件は下記）。 |
| 確認の厳重化 | 「本番に反映」などの文言入力や、対象テーブル・件数表示で誤操作を防げる。 |

### 2.2 実行環境の前提（重要）

- **同期の向き**: ソース＝「今このアプリがつながっている DB」、ターゲット＝本番 DB。
- **本番の接続先は画面で入力する一過性**とする。環境変数（`TARGET_DATABASE_URL`）やファイルには書かず、操作時にだけ入力し、どこにも保存しない。これにより「どこかに書いておく」ことによる漏洩リスクを避ける。
  - 管理画面で「本番マスタ更新」を開く → **本番の接続 URL を入力**（保存されない）→ 「一覧を取得」で検証 → マスタを選択 → 厳重確認 → 実行 → ローカル DB から本番 DB へ選択したマスタだけ upsert。

---

## 3. 確認の厳重化（案）

1. **対象の明示**: 選択したマスタ（テーブル名または論理名）と、それぞれの件数を表示する。
2. **二段階確認**: 「実行」押下でモーダルを出し、「本番に反映する」等の文言を入力しないとボタンを有効にしない。
3. **実行前の注意**: 「本番 DB に直接書き込みます。対象マスタは上書きされます。」などの注意文を常時表示。
4. （任意）実行ログを画面に表示し、どのテーブルが何件更新されたか残す。

---

## 4. 選択式同期の実装方針

- **既存の依存順を崩さない**: `MASTER_DELEGATES_IN_ORDER` はそのまま「全マスタの順序」の正本として使う。
- **選択**: 画面では「同期するマスタ」をチェックボックスで選択する。未選択のテーブルはスキップするが、**実行時は選択されたものだけを、上記順序で**同期する（選択されたもの同士の依存関係は既存順序で満たされる）。
- **一覧の保守**: スキーマでマスタテーブルを追加・削除したときは、**MASTER_DELEGATES_IN_ORDER を 1 箇所だけ**更新する。CLI の `db:sync-masters` と管理画面は**同じリスト・同じ同期ロジック**を参照するようにし、二重管理にしない（例: 同期ロジックを `lib/` か `server/` に共通化し、CLI と Server Action の両方から呼ぶ）。

---

## 5. セキュリティ・認可

- 管理画面であるため、**既存の管理用アカウント（ADMIN_EMAIL 等）の認可**のみで表示・実行可能とする。
- `TARGET_DATABASE_URL` は環境変数で渡す想定。本番デプロイ用の環境には設定しない。

---

## 6. 今後の作業（実装する場合）

1. **同期ロジックの共通化**: `sync-masters-to-target.ts` の「順序リスト＋テーブルごと upsert」を、CLI と Server Action の両方から使えるモジュールに切り出す（例: `src/server/lib/sync-masters-to-target.ts` または `src/lib/sync-masters.ts`）。CLI はそのモジュールを呼ぶだけにする。
2. **Server Action**: 例: `syncMastersToProduction(selectedDelegateNames: string[])`。環境変数 `TARGET_DATABASE_URL` が未設定ならエラー。SOURCE は現在の `DATABASE_URL`（接続中の DB）。
3. **管理画面**: `/dashboard/admin/master-production-sync` のような画面を追加。マスタ一覧（MASTER_DELEGATES_IN_ORDER を人間向けラベル付きで表示）をチェックボックスで表示し、確認文言入力後に Server Action を呼ぶ。
4. **ドキュメント更新**: PRODUCTION_RELEASE_GUIDE.md の「マスタ同期」に「管理画面からも実行可能（ローカルで TARGET_DATABASE_URL を設定した場合）」と追記。

---

## 7. 破棄する「以前の script」について

- 更新で DB の内容・スキーマが変わるたびにスクリプトが合わなくなる問題は、**同期対象テーブル一覧を 1 箇所（MASTER_DELEGATES_IN_ORDER）で管理し、スキーマ変更時にそこだけ手入れする**運用にすれば軽減できる。
- 管理画面から実行する場合も、**同じ一覧・同じロジック**を使うことで、CLI と画面の両方で「スキーマ変更時の更新箇所は 1 つ」にできる。
