## 2. ロードマップのレイヤー構造

- **レイヤーA: コア体験の土台（ゲームサイクル・バランス）**
  - 探索出撃コスト（基本探索キット消費）の仕様確定と実装。
  - 生産チェーン（5設備・基本探索キット組立）の文章と数値を docs 群に反映。
  - 「資源なしでも探索できるが報酬差あり」の設計を採用するかどうかの最終判断。
- **レイヤーB: 遊びやすさ（オンボーディング・ガイド・UI/UX）**
  - 「最初に何をすればいいか」が短く伝わる HowTo／ガイドの整理。
  - 探索UI（ログ・ステータス・ボタン）の負荷ポイント洗い出しと改善。
  - 資源不足時や日次キャップまわりのフィードバック文言・誘導。
- **レイヤーC: コンテンツ & 実験（Later/Next の遊び）**
  - 管理画面から増やせる探索エリア・敵・クエスト・スキルなどの拡張。
  - 市場・PvP・GvG・技能イベント拡張など、将来の「遊びの幅」を広げるアイデア群。

このファイルでは、**1〜3ヶ月のあいだに「どのレイヤーからどこまで進めるか」**を決める。

  - A: 基本探索キットの出撃コスト仕様を spec/docs に落とし込み、文書上の TODO を潰す。
  - A: 生産チェーン（5設備・100回分/日〜200回分/日）の説明を `docs/018` / `docs/019` に反映。
  - B: 「はじめにやること 3 ステップ」の文章草案を `docs/056_how_to_play_guide.md` ベースで整える。
- **フェーズ2（2ヶ月目） — コア実装＋遊びやすさ**
  - A: 探索開始時の資源消費ロジックと UI フィードバック（資源不足メッセージなど）を実装。
  - B: `guide` ページやダッシュボードの導線を整え、HowTo／バトル属性ガイドへの道をわかりやすくする。

## 6. MVP 本リリース前に「まとまって入れたい」中〜大きめテーマ

ここは「MVP テスト公開 → 本番アップデート（データ消去の可能性あり）」までに、**ひとかたまりとして入れておきたい**大きめテーマのメモ。  
1〜3ヶ月ロードマップの外側も含むが、現時点で頭にあるものを一覧にしておく。

- **資源消費機能とその設計（A）**
  - `docs/064_game_cycle_design.md` ベースの **基本探索キット消費・資源なし出撃の扱い** を、spec/実装まで繋げて完結させる。
  - 生産量と「1日あたり本番探索回数」の関係が説明できる状態にする。
- **関わる設備関連（A/C）**
  - 5設備＋派生設備を、「探索資源を増やす設備」「装備寄りの設備」など役割が分かる形で整理。
  - 派生型は DB 上の親子関係に縛られず、**コンテンツ実装時に“横並び”で基本・派生を選べる**設計に寄せていく（FacilityVariant の扱い見直し）。
- **開拓任務の詰め（クエスト報酬・解放）（A/B）**
  - クリア判定だけでなく、**報酬付与（資源・アイテム・通貨等）とコンテンツ解放（設備コスト上限・研究ツリー・探索テーマなど）** の仕組みを作る。
  - 「開拓任務のクリアで設備コスト上限が増える」といったルールを、spec/054 と実装（`src/server/actions/quest.ts` など）に落とす。
- **研究ツリー第二弾まで（A/C）**
  - 研究ツリーの **第1弾 → 第2弾** の解放の仕組みを整える（研究グループ・前提条件・開拓任務との連携）。
  - 「どの研究がどの順番で解放されるか」がプレイヤー視点で追える状態にする。
- **探索テーマ第二弾（1エリア）まで（C）**
  - 探索テーマ・エリアを少なくとも **第二弾 1 エリア分** まで追加し、開拓任務経由で解放される流れを実装。
  - 既存テーマとの難易度差・報酬差が説明できるように設計しておく。
- **市場機能（C・大テーマ）** ※Later から直近実装に移動（§4）
  - 基本探索キットを含む「ほぼすべてのアイテム」が取引可能になるため、**副次効果が非常に大きい**。
  - `docs/ideas/market.md` と `docs/064_game_cycle_design.md` / 経済設計 docs を前提に、  
    - 取引対象、手数料、最低価格・最低出品単位  
    - 通貨を産まない・デフレ寄せ設計との整合  
    を詰めたうえで、実際の UI / API を作る。
- **ガイド周り・遊びやすさ向上（B）**
  - HowTo やバトル属性ガイドに加え、**「次にやること」導線**（開拓任務・研究・探索テーマ）を整理。
  - ログやエラーメッセージ、資源不足時の案内、チャットからの誘導など、プレイ中の「迷子ポイント」を潰していく。
- **設備加速券・特別アイテム（B/C）**
  - **設備画面で消費**し、生産完了を即時または短縮する。**特別アイテム**として実装し、バッグのタブは「特別」（旧・課金）で表示。アイテム名は **緊急製造指示書**。将来は課金アイテムとして同一アイテムを販売する想定（`docs/026`）。
  - HowTo を**開拓任務ベース**で進め、任務クリア報酬で緊急製造指示書を付与し「工業で生産→受け取り→もう一度探索」を待ち時間なしで体験させる。設計は **`docs/065_special_items_and_facility_speed.md`**。
- **通知機能の完成と任務連動（B）**
  - ヘッダーの「通知を表示（準備中）」を有効化し、通知一覧・既読・種別を実装する。
  - **新しい任務を受注したら通知に出す**。プレイヤーが通知から任務画面に飛び、**任務ひとつひとつが HowTo を兼ねる**ので、任務画面で「今やること」と説明を読めるようにする。毎回任務を見に行く導線になるため、そこで説明を載せれば HowTo が伝わりやすい（`docs/065` §4）。
- **装備・ロボパーツのシリーズ追加（C）**
  - 装備品の **2シリーズ目**、ロボパーツの **1シリーズ目** を追加し、ステータス傾向やビルドの幅を増やす。
  - 追加分の作戦・戦闘バランス（攻防・速度・スキル構成）もざっくり調整。
- **遺物まわりの拡張（C/A）**
  - 遺物の効果種類を増やす（戦闘パッシブ以外も含めて、探索・工業・経済に関わる効果も検討）。
  - 効果追加のたびに **戦闘ロジックのどこを拡張するか** を決めておき、無秩序に増えないようルール化したい。
- **遺物・装備の解体とリロール（C）**
  - 遺物・装備を **解体して部品/かけらに戻す仕組み** を入れる（market アイデアでは「部品・欠片のみ売買」前提）。
  - 遺物については、「解体かけらでメイン効果固定・他だけ再抽選（リロール）」のような再挑戦機能を検討。
- **スキル分析書とスキルレベル影響の見直し（A/C）**
  - 現状は分析書でレベルアップできるが、**レベルアップが戦闘にどう効くか** がまだ薄い（または未実装の部分がある）。
  - レベル N→N+1 の効果（威力・CT・付随効果など）を spec/実装で整理し、プレイヤーに分かる形にする。
- **チャット機能まわり（B/C）**
  - **見やすさ・個人設定**（ミュート・フォントサイズ・テーマなど）を整えつつ、
  - **機能拡張**（アイコン表示、名前クリックでユーザページへ飛ぶ等）を追加。
  - 個人チャットは導入したいが、人数が少ないうちは諸刃になりやすいので **後ろ目** に置き、まずは全体チャットの質向上を優先。
- **郵便機能（C）** ※市場機能とは別テーマ
  - 運営からの告知・補填・プレゼント配布などを、ゲーム内メールで届けられるようにする。**市場（取引）とは無関係に、運営配布・告知用として必要。**
  - 将来的なプレイヤー間送付は、経済やサブ垢問題とセットで慎重に検討。
- **運営用ログ・メトリクス（A/B）**
  - 探索回数・勝率・平均報酬・所要時間、工業の生産/受け取り状況、クエスト達成状況、チャット利用など、**ゲーム運営に必要な指標**を整理する。
  - どこまでを **アプリ内ログ（DBテーブル）** とし、どこからを **外部メトリクス（例: GA4 など）** に任せるか方針を決める。
  - MVP 本リリース前に「最低限これだけあればゲームの健康状態を追える」というログ/メトリクス一式を仕込んでおく。
- **全体のセキュリティチェック（A/B）**
  - 認証・セッション・管理画面の突入条件・API の認可・環境変数や秘密情報の扱いなど、**全体のセキュリティをどこかのタイミングでちゃんとチェックする**ことをロードマップに含めておく。
  - 本リリース前または定期的に、チェックリストや `manage/SECURITY_READINESS.md` 等を参照して見直す想定。

これらは「すべて 1〜3ヶ月でやる」というより、**MVP テスト版から本リリース版へアップデートするまでにどこまで入れるか**を決めるための候補リスト。  
優先度や順番は、プレイヤーの反応・開発コスト・バランス調整の負荷を見ながら、直近TODOや次期ロードマップで細かく切っていく。
