# レベルキャップとキャップ到達後の経験値→アイテム付与

ゲーム全体の**レベルキャップ**を設け、キャップ到達後は**レベルは上げず**、代わりに「レベル 1 分の経験値が貯まるごとに振り直しアイテムを 1 個付与する」方式にする設計。  
現状のレベルアップ仕組みからの変更点を整理する。

- **前提**: ステータス振り直しアイテムは未実装。本 doc では「キャップ到達後に経験値の代わりに付与する報酬」として同じアイテムを想定する。
- **キャップ解放**: 将来レベルキャップを引き上げる想定はあるが、現時点では**レベルはキャップで止まり、上がらない**。
- **依存**: `docs/048_experience_and_levelup.md`、`spec/048_level_and_status_allocation.md`、`docs/09_character_implementation_notes.md`。

---

## 1. 現状のレベルアップの流れ（変更前）

### 1.1 データ・計算

- **Character**: `level`, `experiencePoints`, `CAP`, 7 種基礎ステ（STR/INT/VIT/WIS/DEX/AGI/LUK）
- **累計経験値 → レベル**: `computeLevelFromTotalExp(totalExp)`（`src/lib/level.ts`）
  - Lv N に達する累計 = `10 × (N-1)×N / 2`
  - Lv N → N+1 に必要な経験値 = **N × 10**
- **レベル → CAP**: `CAP(N) = 560 + 60×(N-1)`（`getCapForLevel(level)`）
- **レベルアップ時のステ**: 各ステに `floor(deltaCAP×0.10)` を加算。残り 30% は未割り振りとしてプレイヤーが後で割り振る（spec/048 §2.6）。

### 1.2 経験値付与

- **API**: `grantCharacterExp(userId, characterIds, amount, tx?)`（`src/server/actions/character-exp.ts`）
- **流れ**:
  1. 対象キャラの `experiencePoints` に `amount` を加算
  2. 加算後の累計から `newLevel = computeLevelFromTotalExp(newExp)` を計算
  3. `newLevel > oldLevel` ならレベルアップ処理: `level`, `CAP`, 7 ステを更新（2.6 に従う）
  4. レベルが上がらない場合は `experiencePoints` のみ更新

- **呼び出し元**: 主に探索報酬（`finishExploration` 内で `grantCharacterExp` を呼ぶ）

### 1.3 キャップの有無

- 現状は**レベル上限なし**。経験値が足りればどこまでもレベルが上がる。

## 2. 導入する仕様

### 2.1 ゲーム全体のレベルキャップ・段階解放

- **定数**: ゲーム全体で「最大レベル」を 1 つ定義する（**LEVEL_CAP**）。**段階的に解放**する想定（例: 初期 30 → 40 → … → 100）。定数変更＋リリースで対応。
- **意味**:
  - 経験値付与で計算上のレベルがキャップを超える場合でも、**レベルはキャップで止める（それ以上は上がらない）**。
  - キャップ 30 のとき: Lv29→30 は 290 exp（通常式）。30→31 は「まだ解放されていない」ので、超過分はアイテムに変換する。

### 2.2 キャップ到達後の扱い：「次のレベル分」を固定 1000 にしたアイテム付与

- **キャップに到達したあと**（例: Lv30 や Lv100）、さらに経験値が入った場合:
  - **レベルは上げない**。
  - 代わりに「**次のレベル分の経験値**」が貯まるごとに、**振り直しアイテムを 1 個ユーザーに付与**する。
- **「次のレベル分」は常に 1000（固定）**: **EXP_PER_LEVEL_AT_CAP = 1000**。キャップが 30 でも 100 でも「1000 貯まったら 1 個」に統一する。解放されていない先のレベル（31, 32, …）を「1000 = 1 アイテム」で扱うため。Lv29→30 の 290 は通常どおりで、30→31 相当から 1000 単位になる。
- **処理イメージ**:
  - キャラがすでに Lv100（キャップ）である。
  - 経験値が加算される（探索報酬など）。
  - 加算後の累計が「キャップ到達に必要な経験値」を超えた分（オーバーフロー）を、「レベル 1 分」で割る → 商の数だけ振り直しアイテムを付与。
  - 余り（端数 exp）はキャラの累計経験値として保持する（次に経験値が入ったときに加算して再計算）。
- **付与先**: アイテムは**ユーザー（userId）の所持**に加える（キャラに紐づけず、バッグなどに 1 個ずつ付与）。対象は「経験値を得たキャラのオーナー」でよい。

### 2.3 具体例（LEVEL_CAP = 30 または 100 の場合）

- キャップ 30 のとき: キャップ到達に必要な累計 = `getRequiredExpForLevel(30)` = 4,350。Lv30 のキャラに 2500 exp 入ったとするとオーバーフロー = 2,500。「1 レベル分」= **EXP_PER_LEVEL_AT_CAP = 1,000**（固定）。付与個数 = floor(2,500 / 1,000) = **2 個**。
- キャップ 100 のとき: 同様にオーバーフローを 1,000 で割った商が付与個数。
- **余りを保持する場合**: 余り 500 を「次回用」に残すなら、`experiencePoints` を 49,500 + 500 = **50,000** に更新する。
- **余りを保持しない場合（推奨・下記 2.4）**: アイテム付与後は常に `experiencePoints = getRequiredExpForLevel(LEVEL_CAP)` にリセットする。余りは捨てる。このとき**テーブルは変更不要**で、処理だけ変えればよい。

### 2.4 「累計が多くても Lv100 なら必要経験値は 1,000」にしたい場合（余りを保持しない）

- **希望**: レベルキャップ中は「次のレベルに必要な経験値」を常に **LEVEL_CAP × 10**（例: 1,000）と統一したい。キャップ解放後も「Lv100 の人はみなあと 1,000 で 101」にしたい。
- **やり方**: キャップ到達後にアイテムを付与したら、**余りを保持せず** `experiencePoints = getRequiredExpForLevel(LEVEL_CAP)` に更新する。  
  - アイテム付与後の更新を `getRequiredExpForLevel(LEVEL_CAP) + (overflow % expPerLevelAtCap)` ではなく、**`getRequiredExpForLevel(LEVEL_CAP)` のみ**にする。
- **テーブル**: 変更不要。`Character.experiencePoints` のまま。
- **処理**: `grantCharacterExp` 内の「キャップ超過時の experiencePoints 更新」を上記に変えるだけ。
- **結果**: Lv100 のキャラは全員 `experiencePoints = 49,500`。表示も「あと 1,000 で Lv101」で揃う。キャップ解放後も全員あと 1,000 で 101 になる。
- **トレードオフ**: オーバーフローの端数（最大 999）は毎回捨てるため、その分の「次のレベルへの進捗」は付かない。代わりに説明が単純になり、UI も「必要 1,000」で統一できる。現状は 1 回の経験値付与が最大でも十数程度のため、余り廃棄の実質的な影響は小さい（許容範囲）。

### 2.5 累計経験値の別案（参考）

- **余りを保持する案**: 保存を `getRequiredExpForLevel(LEVEL_CAP) + (overflow % expPerLevelAtCap)` にすると、Lv100 の累計は 49,500～50,499。キャップ解放後は「あと 1～1,000」で人によって違う。
- **累計をどこまでも増やす案**: 別カラムで「アイテムに換算済み量」を管理すれば累計 70,000 の Lv100 も表現できるが、キャップ解放時に 70,000 なら即 Lv101 になる。「必要 1,000 で統一」とは相性が悪い。

### 2.6 レベルキャップ解放時（例: 30→40、40→100）の挙動

- **解放手順**: LEVEL_CAP 定数を 30 → 40 などに変更してリリースするだけ。DB 変更不要。
- **次回の経験値付与時**: キャラは Lv30・experiencePoints = 4,350 のまま。ここに amount が加算され、`newLevel = computeLevelFromTotalExp(4350 + amount)`、`effectiveNewLevel = min(newLevel, 40)` でレベルが決まる。
  - **1000 だけ入った場合**: 5,350 → Lv31。**30→31 の 1 レベルアップ**。
  - **5000 一気に入った場合**: 9,350 → Lv40 相当以上。**30→40 の複数レベルアップ**になる。
- つまり「1000 判定しかしていない」のではなく、**その回の付与量が大きければ一気に複数レベルアップ**する。表示は「あと 1000 で次のレベル」（EXP_PER_LEVEL_AT_CAP 固定）で統一できる。

## 3. 変更が必要な箇所

### 3.1 定数・設定（レベルキャップはプログラムに集約）

| 箇所 | 内容 |
|------|------|
| **`src/lib/constants/level.ts`** | **LEVEL_CAP**（段階解放なら例: 30 → 40 → 100）、**EXP_PER_LEVEL_AT_CAP**（固定 1000）、**LEVEL_CAP_REWARD_ITEM_CODE**。 |

- **定数は一箇所に集約する**。既存の `src/lib/constants/`（currency, companion, relic など）と同様、レベルまわりの定数も constants 配下にまとめると見通しがよい。
- **レベルキャップはプログラムの定数として持つ**（DB やマスタには載せない）。キャップを上げる頻度は非常に間隔が長い想定のため、定数変更＋通常リリースで十分。
- **ビルド時**: 定数はコンパイル時にインライン化されるため、参照のたびの負荷は気にしなくてよい（実行時コストは実質変わらない）。
- 一般的なゲームでも、最大レベルやキャップのような「めったに変えない値」はコードに組み込んでいることが多い。DB に持つと経験値付与のたびに読みに行く処理負荷が増えるだけで、変更頻度の低さを考えるとメリットが小さい。
- 将来「運営がマスタでキャップを変えたい」など要件が変わったら、その時点で設定テーブルや環境変数に移す選択肢はある。

### 3.2 経験値付与（grantCharacterExp）

| 変更内容 | 詳細 |
|----------|------|
| レベルをキャップで止める | 計算された `newLevel` が LEVEL_CAP を超える場合は、レベルは LEVEL_CAP とする。CAP・ステ更新も LEVEL_CAP 相当で一度だけ行えばよい（すでにキャップにいる場合はレベルアップ処理は行わない）。 |
| キャップ到達後の経験値 | すでに `level === LEVEL_CAP` のキャラに経験値が入った場合: 累計を加算したうえで、オーバーフロー = `newExp - getRequiredExpForLevel(LEVEL_CAP)` を計算。`itemsToGrant = floor(overflow / EXP_PER_LEVEL_AT_CAP)`（EXP_PER_LEVEL_AT_CAP = 1000 固定）。ユーザーに振り直しアイテムを `itemsToGrant` 個付与。**余りを保持しない**: `experiencePoints = getRequiredExpForLevel(LEVEL_CAP)` に更新。 |
| キャップ未満でキャップを超える場合 | 加算後 `newLevel > LEVEL_CAP` になる場合: まず LEVEL_CAP までレベルアップ処理（1 回だけ CAP・ステ更新）。その後、残り経験値でオーバーフローを計算し、上記と同様にアイテム付与と `experiencePoints` の更新（余りを保持しないなら `getRequiredExpForLevel(LEVEL_CAP)` に統一）を行う。 |

### 3.3 レベル計算ユーティリティ（lib/level.ts）

| 変更内容 | 詳細 |
|----------|------|
| キャップ用の補助 | キャップ超過時の「1 レベル分」は **EXP_PER_LEVEL_AT_CAP**（1000 固定）。`LEVEL_CAP * 10` は使わない。 |
| 既存関数 | `computeLevelFromTotalExp` は理論レベル用のまま。付与側で `min(computedLevel, LEVEL_CAP)` とオーバーフロー計算を行う。 |

### 3.4 アイテム付与

| 項目 | 内容 |
|------|------|
| 付与するアイテム | 振り直し用アイテムの **Item.code** を定数で持つ（`LEVEL_CAP_REWARD_ITEM_CODE = "reconstitution_ampoule_alpha"`、表示名: 再構築アンプルα）。 |
| 付与先 | 経験値を得たキャラの `userId` に対するユーザー所持（UserItem や所持品テーブル）に加える。 |
| タイミング | `grantCharacterExp` 内で、キャップ超過分を計算した直後に、同一トランザクションでアイテム付与を行う。 |

- 付与対象アイテムは code `reconstitution_ampoule_alpha`（再構築アンプルα）。定数は `src/lib/constants/level.ts` の `LEVEL_CAP_REWARD_ITEM_CODE`。

### 3.5 呼び出し元の影響

| 呼び出し元 | 変更 |
|------------|------|
| `finishExploration` 内の `grantCharacterExp` | 変更不要。経験値付与のみ。内部でキャップとアイテム付与が行われる。 |
| その他 `grantCharacterExp` を呼ぶ場所 | 同上。 |

## 4. データ・仕様の整理

### 4.1 レベルキャップ定数

- **LEVEL_CAP**: 整数（例: 100）。この値が最大レベル。経験値でこれ以上は上がらない。

### 4.2 キャップ超過時付与アイテム

- 振り直し用アイテムの **Item.code** を定数 `LEVEL_CAP_REWARD_ITEM_CODE` で持つ。現在の値: `reconstitution_ampoule_alpha`（再構築アンプルα）。
- 付与は `grantStackableItem`（`src/server/lib/inventory.ts`）を使う。code で Item を検索して id を取得してから呼ぶ。

### 4.3 表示・UX

- キャラが Lv100（キャップ）のとき、「これ以上レベルは上がりません。経験値は振り直しアイテムに変わります」などと案内できる。
- 探索報酬画面などで「〇〇の経験値の代わりに振り直しアイテムを N 個獲得」と出してもよい。

## 6. 実装フェーズ（案）

1. **Phase 1**: `LEVEL_CAP` 定数の追加と、`grantCharacterExp` 内でレベルを `min(newLevel, LEVEL_CAP)` で打ち止めする。キャップ超過時の `experiencePoints` の扱い（オーバーフロー計算・余りの繰り越し）まで実装する。
2. **Phase 2**: 振り直しアイテムのマスタ定義（または仮 ID）と、ユーザーへのアイテム付与処理。`grantCharacterExp` から「キャップ超過分 → アイテム N 個付与」を呼び出す。
3. **Phase 3**: 探索報酬などで「振り直しアイテムを N 個獲得」と表示する UI（任意）。
4. **Phase 4**: 振り直しアイテムの本来の用途（ステータス振り直し）を実装する。

## 7. まとめ

| 項目 | 現状 | 変更後 |
|------|------|--------|
| レベル上限 | なし | ゲーム全体で LEVEL_CAP（例: 100）。**レベルはこれ以上上がらない**。 |
| 経験値付与（キャップ未満） | 経験値でレベルアップ | 従来どおり。キャップまでレベルアップする。 |
| 経験値付与（キャップ到達後） | （存在しない） | 経験値を加算するがレベルは上げない。**レベル 1 分の経験値が貯まるごとに振り直しアイテムを 1 個ユーザーに付与**。余り exp は繰り越す。 |
| キャップの将来 | - | キャップ値の引き上げは想定しているが、現仕様ではレベルはキャップで固定。 |

この変更により、「レベル 100 になったあとはレベルは上がらないが、レベル 1 分の経験値が貯まるごとに代わりに振り直しアイテムをあげる」仕様を満たせる。
