# 019: 生産判定・受け取り仕様

設備の**生産量**と**受け取りタイミング**の判定ルールを定義する。  
「最終受け取り日時」と「受け取りボタン押下日時」の間の経過時間で実行可能なサイクル数を計算し、その回数分だけ受け取る形式とする。

---

## 1. 方針

- **受け取りタイミング**：プレイヤーが**受け取りボタンを押した時**にのみ受け取り処理を実行する。バックグラウンドで自動付与しない。
- **製造量の判定**：**最終受け取り日時**から**受け取りボタン押下日時（現在時刻）**までの経過時間を用い、レシピの**周期（cycleMinutes）**で割り、**実行できた回数（サイクル数）**を計算する。その回数分の出力を付与し、入力があれば消費する。
- **初回の「最終受け取り」**：
  - **初期エリアの設備**：アカウント作成日時（User.createdAt）を「最終受け取り」として扱う。
  - **新規に配置した設備**：設備配置日時（FacilityInstance.createdAt）を「最終受け取り」として扱う。

→ 設備が存在しなかった期間の生産は発生しない。

---

## 2. 用語

| 用語 | 説明 |
|------|------|
| **最終受け取り日時（lastReceivedAt）** | その設備インスタンスで「最後に受け取りを実行した時刻」。この時刻以降の経過時間が今回の生産計算の対象になる。 |
| **受け取り** | プレイヤーがボタン操作で「この設備の生産物を受け取る」と明示したときの処理。経過時間とレシピからサイクル数を算出し、入力消費・出力付与・lastReceivedAt 更新を行う。 |
| **サイクル** | レシピの 1 回分の生産（周期 cycleMinutes で 1 回。入力消費・出力 1 回分）。 |

---

## 3. 計算ルール

### 3.1 サイクル数の算出

- **経過分数** = 受け取りボタン押下日時（現在時刻） − 最終受け取り日時（分単位で計算。端数は切り捨てまたは秒を分に換算してよい）。
- **経過分数にキャップを適用**：経過分数が **24 時間（1440 分）を超える場合は 1440 分として扱う**（§5 生産キャップ）。
- **実行可能サイクル数** = floor( キャップ適用後の経過分数 / レシピの cycleMinutes )  
  ※ 0 以上整数。経過が周期未満なら 0。

例：最終受け取りが 10:00、受け取り押下が 10:35、周期 20 分 → 経過 35 分、35/20 = 1 余り 15 → **1 サイクル**。  
次回の「最終受け取り」は 10:20 に更新する（10:00 + 1×20）ので、残り 15 分は次回に繰り越される。

### 3.2 入力がある設備の場合

- レシピに**入力**（RecipeInput）がある設備は、**所持在庫**で実行できる回数に上限がある。
- **実際に実行するサイクル数** = min( 時間ベースのサイクル数, 入力素材で実行可能な回数 )  
  - 入力素材ごとに「所持数 / 1回あたり消費数」の切り捨てを求め、その最小値が「入力で実行可能な回数」。
- 実行したサイクル数ぶんだけ**入力消費**と**出力付与**を行い、**lastReceivedAt** を「直前の lastReceivedAt + 実行サイクル数 × cycleMinutes」に更新する。
- 入力不足で時間ベースより少ない回数しか実行しない場合でも、**使った時間分だけ lastReceivedAt を進める**。未実行分の時間は次回受け取り時に再度計算される（繰り越し）。

例：浄水施設（20分、水200→飲料水100）。経過で 3 サイクル可能だが水が 300 しかない → 1 サイクルだけ実行。水 200 消費、飲料水 100 付与、lastReceivedAt を +20 分進める。

### 3.3 初回の lastReceivedAt

- **初期エリアの 5 設備**：受け取り履歴が無いときは **User.createdAt** を「最終受け取り日時」として使う（アカウント作成時から生産が始まっているとみなす）。
- **その後に配置した設備**（追加解放エリアで建設した設備）：受け取り履歴が無いときは **FacilityInstance.createdAt** を「最終受け取り日時」として使う（配置した時点から生産開始）。

※ 初期エリアかどうかは、設置エリアの code='initial' か、または「強制配置でユーザーが建設していない」かで判別する。実装では FacilityInstance に lastReceivedAt を持たせ、NULL のとき上記のフォールバック日時を使う形にするとよい。

---

## 4. データまわり（実装時の前提）

- **FacilityInstance** に **lastReceivedAt**（DateTime, NULL 可）を持つ。  
  NULL = まだ一度も受け取りしていない。計算時は NULL なら初回として User.createdAt または FacilityInstance.createdAt を使う。
- **在庫**：ユーザーごとの素材・製品の所持数が必要。例：UserInventory（userId, itemId, quantity）。受け取り時に出力を加算し、入力がある設備では入力分を減算する。
- 受け取り処理は**トランザクション**で行う（在庫の増減・lastReceivedAt 更新を一括でコミット）。

---

## 5. オフライン放置の扱い（生産キャップ）

- **生産キャップは一律 24 時間分**とする。最終受け取り日時から受け取りボタン押下までに 24 時間を超える経過がある場合、**計算に使う経過時間は最大 24 時間（1440 分）**として扱う。
- 例：48 時間経過していても、1440 分 / cycleMinutes で算出したサイクル数までしか受け取りできない。それ以上は付与しない。
- ひとまずこの値で固定。将来必要ならキャップ値を設定可能にする。

---

## 6. まとめ

- **受け取り** = ボタン押下時にのみ実行。
- **製造量** = 最終受け取り日時〜現在の経過で **floor(経過分 / 周期)** サイクル。入力あり設備は在庫で実行可能な回数にキャップ。
- **初回** = 初期エリアはアカウント作成時、新規配置は設備配置時を「最終受け取り」とみなす。
- **更新** = 受け取り後に lastReceivedAt を「前回 + 実行サイクル数 × 周期」に更新し、端数時間は次回に繰り越す。
- **生産キャップ** = 経過時間は最大 24 時間分まで。それ以上は受け取り対象にしない。

以上を生産判定・受け取りの基準とする。
