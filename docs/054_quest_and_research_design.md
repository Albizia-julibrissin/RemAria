# クエストと研究の設計（定義・前提・作業分解）

**実装に必要な要点は §2 用語と定義・§3 前提と方針。** 正式仕様は `spec/054_quests.md`。詳細は以下。

---

研究メニューとクエスト機能の関係を整理し、実装のための定義と作業分解をまとめる。  
本 doc は「何を決めるか」「どう分解するか」の設計メモ。実装状況は manage/MVP_PROGRESS.md 参照。

---

## 1. 全体像

- **研究**：設備の基本型解放・クラフトレシピ解放・（将来）設備派生型解放を行うメニュー（047 で定義済み。条件は「別 spec または仮条件」とされている）。
- **クエスト**：ストーリー・研究・一般（デイリー等）の 3 種を想定。**研究クエスト**をクリアすると**研究ポイント**が付与され、そのポイント＋アイテム消費でクラフトレシピや設備のベース型を解放する。
- **称号**：クエスト報酬の一つとして付与することがある（プロフィール表示は MVP で未実装）。

```
[クエスト] ── ストーリークエスト ── 一度限り、主に進行・演出
         ├── 研究クエスト ─────── 一度限り → 研究ポイント付与
         └── 一般クエスト ─────── デイリー等、繰り返し可

[研究ポイント] ＋ [アイテム消費] → 研究メニューで「解放」に使用
  - クラフトレシピ解放（UserCraftRecipeUnlock）
  - 設備ベース型解放（UserFacilityTypeUnlock）
```

## 2. 用語と定義

### 2.1 クエストの種類

| 種別 | 説明 | 受注 | 繰り返し | 主な報酬 |
|------|------|------|----------|----------|
| **ストーリークエスト** | ストーリー進行用。前提クエスト・章立てを想定。 | 要検討（下記 3.1） | 一度限り | 進行フラグ・次のクエスト解放・演出・場合により称号 |
| **研究クエスト** | 研究ポイント獲得用。クリアでポイント付与。 | 要検討 | 一度限り | 研究ポイント・場合により称号 |
| **一般クエスト** | デイリー・ウィークリー等。 | 要検討 | 繰り返し可 | 資源・経験値・お金・場合により称号 |

### 2.2 研究まわり

- **研究ポイント**：ユーザーが所持する「研究で解放に使う通貨」の一種。研究クエストクリアで付与。整数で保持（User または専用テーブル）。
- **解放**：研究メニューで「研究ポイント ＋ 指定アイテム消費」により行う。
  - **設備のベース型**：UserFacilityTypeUnlock に 1 行追加（047 既存）。
  - **クラフトレシピ**：UserCraftRecipeUnlock に 1 行追加（046 で「将来」とされている）。
- **設計図**：026 では「設計図を研究メニューで消費して解放」とある。本設計では「研究ポイント ＋ アイテム」を第一とし、設計図は「そのアイテムの一種」として扱う（同一メニューで両方使える形にしてもよい）。

### 2.3 クエストの達成条件（達成タイプ）

**初期想定**の 4 種は §3.5 にまとめた。以下は一覧と将来候補。

| 達成タイプ | 説明 | 計上元 | 初期 |
|------------|------|--------|------|
| 技能イベント N 回 | 探索中に技能判定を成功させた回数 | 探索終了時に skillSuccessCount を進捗に加算 | ◎ |
| 特定装備クラフト N 回 | 指定 EquipmentType で N 個クラフト | executeCraft で装備作成時、該当なら進捗加算 | ◎ |
| 特定敵 N 体 | 指定 Enemy / EnemyGroup を N 体倒す | 探索・練習戦闘終了時に倒した敵を進捗に加算 | ◎ |
| 特定エリア N 回 | 指定 ExplorationArea を N 回クリア | 探索 finish で該当 areaId なら進捗加算 | ◎ |
| お金を N 消費 | プレイヤー市場等（市場未実装） | 市場実装時に計上 | — |
| 特定アイテムを N 個所持 | 所持数が N 以上で達成 | 所持が変わる処理のあとでチェック | — |
| ストーリー前提 | 特定クエスト完了済みで出現 | prerequisiteQuestId 等で参照 | 使用 |

その他：**特定施設を 1 個建設**なども追加可能。

### 2.4 称号

- クエスト報酬として**称号**を付与することがある。
- 称号はユーザー単位で「所持一覧」と「装備中 1 つ」を持つ想定（MVP ではプロフィール表示未実装のため、データだけ用意してもよい）。

## 3. 前提・方針（決めること）

### 3.1 受注の仕方

- **オート受注**：条件を満たした時点で自動で「受注済み」になり、進捗だけが貯まっていく。受注 UI は「一覧・進捗表示」中心。
- **手動受注**：クエスト一覧で「受注する」を押すと受注。前提（他クエスト完了・ストーリー進行）は受注可能条件として表示。
- **ハイブリッド**：ストーリー・研究は「解放された時点で自動受注」、一般は「一覧から手動受注」など。

**方針（確定）**：**ストーリー・研究はオートで作る**。前提クエスト完了で次のクエストが出現し、出現＝受注扱い（受注 API 不要）。**デイリー・ウィークリー等の一般クエストは後で追加するときに手動受注を検討**する。

### 3.2 研究クエストと研究メニューのつながり

1. 研究クエストを 1 本クリア → **研究ポイント**が所持に加算される。
2. 研究メニューでは「解放したい項目」を選び、**必要研究ポイント ＋ 必要アイテム**を消費して解放する。
3. 解放対象は「設備のベース型」（047 の UserFacilityTypeUnlock）と「クラフトレシピ」（UserCraftRecipeUnlock）の 2 種を想定。

これにより「研究メニュー」は**解放**（ポイント＋アイテム消費で設備型・レシピを解放）を提供する。クエスト一覧は下記のとおりクエスト画面に集約する。

### 3.3 クエスト画面とフィルタ

- **クエストはすべて基本のクエスト画面で見られる**。一覧でストーリー／研究（および将来は一般）を**フィルタ**で切り替える。
- 研究クエストの進捗も同じ画面の「研究」フィルタで表示。研究メニュー（解放用画面）とは別で、研究メニューでは解放対象・コスト・実行のみ扱う。

### 3.4 ストーリーと研究クエストの出現

- **最初のストーリークエスト**：達成条件は**「探索を 1 回クリア」**（探索 finish で 1 本クリア）。
- **研究クエストの出現**：**最初のストーリークエストをクリアしたら**研究クエストが出現する（前提クエスト = そのストーリークエスト 1 本）。

### 3.5 初期にサポートする達成タイプ

ストーリー・研究で使う達成タイプは以下 4 種を**初期想定**とする。

| 達成タイプ | 説明 | 計上元 |
|------------|------|--------|
| 技能イベント N 回 | 探索中に技能判定を成功させた回数 | 探索終了時に skillSuccessCount 等を進捗に加算 |
| 特定装備クラフト N 回 | 指定した装備種別（EquipmentType）を N 個クラフト | executeCraft で装備作成時に、該当レシピなら進捗加算 |
| 特定敵 N 体 | 指定した敵（Enemy または EnemyGroup）を N 体倒す | 探索戦闘終了時（＋練習戦闘終了時）に倒した敵を進捗に加算 |
| 特定エリア N 回 | 指定した探索エリアを N 回クリア | 探索 finish で該当 areaId なら進捗加算 |

### 3.6 解放の初期状態と未解放の見せ方

- **解放の初期状態**：**現状 seed で解放している設備・レシピはそのまま初期解放**とする。**研究ポイントは「その先」の設備・レシピ解放に使う**（追加で何を実装するかは未定）。
- **未解放クラフトレシピ**：クラフト一覧では**未解放は出さない**（見せない）。**研究画面**を作ったときに、**研究画面側で「解放可能なレシピ・必要コスト」として表示**する。

### 3.7 称号

- **ストーリー・研究の報酬に称号を入れる**。そのため Title マスタ・UserTitle（所持）・クエストクリア時の称号付与を実装する。装備中表示はプロフィール等で後からでもよい。

### 3.8 クエストの説明文とクリア報告時の演出

- **説明文**：クエストには**説明文（description）**を持たせ、プレイヤーが**経緯・状況**を把握できるようにする。クエスト一覧や詳細で表示する。
- **クリア報告時のメッセージ**：クエスト達成時に**クリア報告**（報酬付与・状態更新）のタイミングで**専用メッセージ**を出し、そこで**ストーリーの演出**を行う。Quest マスタに「クリア報告メッセージ」（clearReportMessage 等）を用意し、報告 UI で表示する。ストーリークエストでは特にこのメッセージで物語の区切りや次の展開を示す。

## 4. データ構造（案）

実装時に spec で確定する。ここでは概要のみ。

### 4.1 クエスト

- **Quest**（マスタ）：id, code, questType（story / research / general）, name, **description**（説明文。経緯・状況がわかるテキスト）, **clearReportMessage**（クリア報告時に表示するメッセージ。ストーリー演出用。NULL 可）, 前提クエスト ID、達成条件（種別＋パラメータ）、報酬（研究ポイント、アイテム、称号など）。繰り返し可否。
- **UserQuest**（進行）：userId, questId, state（available / in_progress / completed）, progress（JSON または別テーブルでカウンタ）, completedAt。デイリー等はリセット用の period を持つ場合あり。

### 4.2 研究ポイント

- **UserResearchPoint**：userId, amount（整数）。または User に researchPoint カラムを追加。

### 4.3 解放コスト（研究メニューで使う）

- 設備ベース型解放：**FacilityTypeUnlockCost**（または既存テーブル拡張）：facilityTypeId, requiredResearchPoint, 必要アイテム（ItemId + amount）のリスト。
- クラフトレシピ解放：**CraftRecipeUnlockCost**：craftRecipeId, requiredResearchPoint, 必要アイテムのリスト。

既存の UserFacilityTypeUnlock / 将来の UserCraftRecipeUnlock は「解放済み」の記録。コストはマスタで持つ。

### 4.3.1 研究グループ（第一弾：錆びれた森林研究）

- **研究グループ**：解放対象をまとめる単位。テーマと直接連携はしないが、実装タイミングを合わせて第一弾は「錆びれた森林研究」とする。
- **ResearchGroup**：id, code, name, displayOrder, **prerequisiteGroupId**（null＝最初のグループ。次のグループは「前提グループの派生型以外をすべてクリア」で解放）。
- **ResearchGroupItem**：researchGroupId, **targetType**（facility_type / craft_recipe）, **targetId**（FacilityType.id または CraftRecipe.id）, **isVariant**（派生型なら true。グループクリア判定では「派生型以外すべて解放」で次グループ解放）, displayOrder。
- **ResearchUnlockCost**：targetType, targetId, itemId, amount。解放時に消費するアイテム（第一弾はアイテム消費のみ。研究ポイントは任意で後から追加可）。
- **グループ解放条件**：あるグループが「利用可能」になるのは、prerequisiteGroupId が null であるか、前提グループに属する**派生型以外の全項目**がユーザーによって解放済みのとき。次の研究グループを解放するには、錆びれた森林研究を**派生型以外すべてクリア**する。

### 4.4 称号

- **Title**（マスタ）：id, code, name, 説明。
- **UserTitle**：userId, titleId, acquiredAt。装備中は User.equippedTitleId など。

### 実装の着手順（推奨）

**「最初のストーリークエスト（探索1回）→ クリア報告 → 研究クエスト出現」まで動かす**ことを第一のマイルストーンにすると、依存が少ない順は以下。

| 順 | タスク | 理由 |
|----|--------|------|
| **1** | **B1** Quest / UserQuest スキーマ | 他に依存しない。Quest マスタに「探索1回」「説明文・clearReportMessage」と、研究クエスト 1 本を seed で用意。 |
| **2** | **B2** 進捗更新・達成判定・報酬付与 | 進捗加算・達成時に報酬（次のクエスト解放＝UserQuest INSERT）。研究ポイント報酬は A1 後に有効にすればよい。 |
| **3** | **B3** 達成タイプ「特定エリア N 回」 | 探索 finish で該当 areaId のクエストに進捗加算・達成判定。これで「探索1回」が動く。 |
| **4** | **D1** 前提クエスト＝出現（オート受注） | ログイン時やクエスト一覧取得時に「前提クエスト完了済みで UserQuest が無いクエスト」を UserQuest に INSERT。最初のストーリー 1 本は最初から 1 件 INSERT。 |
| **5** | **B4** クエスト画面 UI | 一覧・フィルタ（ストーリー／研究）・説明文・進捗・クリア報告メッセージ表示。 |
| 6 | **A1** 研究ポイント所持 | User に researchPoint。研究クエストの報酬で加算するために必要。 |
| 7 | C1（報酬＝研究ポイント） | B2 の報酬種別に researchPoint を追加。実質 B2 拡張。 |
| 8 | B3 残り達成タイプ | 技能 N 回・特定装備クラフト N 回・特定敵 N 体。研究クエストの条件を増やすときに実装。 |
| 9 | A2～A4（解放コスト・解放 API・getResearchState 拡張） | 「その先の設備・レシピ」を決めてからでよい。現状は初期解放のまま。 |
| 10 | D3 称号 | 報酬に称号を入れたいタイミングで Title / UserTitle と付与を実装。 |

**結論**：**B1 → B2 → B3（エリア N 回だけ）→ D1 → B4** の順で進めると、「探索1回クリア → クリア報告メッセージ → 研究クエスト出現」まで一通り動く。そのあと A1 と研究クエスト報酬でポイントを貯め、A2～A4 は解放対象を決めてからでよい。

## 7. まとめ

- **クエスト画面**：全クエストを 1 つのクエスト画面で表示し、フィルタでストーリー／研究（／将来は一般）を切り替える。
- **最初のストーリークエスト**：探索を 1 回クリア。クリアで研究クエストが出現。
- **研究クエスト**：最初のストーリークエストクリア後に出現。クリアで研究ポイント付与。
- **研究メニュー**：解放済み一覧＋「研究ポイント＋アイテム消費で解放」。未解放の設備・レシピと必要コストは研究画面で表示。クラフト一覧では未解放レシピは出さない。
- **解放**：現状の初期解放はそのまま。研究ポイントはその先の設備・レシピ用（追加内容は未定）。
- **受注**：ストーリー・研究はオート（出現＝受注）。手動はデイリー・ウィークリー追加時に検討。
- **達成タイプ（初期）**：技能イベント N 回・特定装備クラフト N 回・特定敵 N 体・特定エリア N 回。
- **クエスト説明文・クリア報告メッセージ**：説明文で経緯を表示。クリア報告時に専用メッセージを出してストーリー演出を行う（Quest.description / Quest.clearReportMessage）。
- **称号**：ストーリー・研究の報酬に含める。Title/UserTitle を実装する（Phase D3）。
