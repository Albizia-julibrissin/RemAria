# 017 spec 作成前チェックリスト

017 を「このままいけるか」確認するためのメモ。spec 作成・実装時に参照する。

## 結論：このまま spec 作成で問題なし

017 の設計は一貫しており、他 doc（08, 15, 02）とも用語が揃っている。  
以下の「未実装・未定義」は **017 実装時に必要になる前提** として、spec や 08 の「将来拡張」に書いておけばよい。

## 2. spec に書いておくとよいこと

- **017 のスコープ**  
  「設備の型（基本型／派生型）と建設時の資源消費。派生型解放は設計図消費（026）。建設は型ごとの建設レシピで資源を消費（希少部品必須ではない）。Item・FacilityInstance・型マスタ・建設レシピ・研究解放・探索報酬の詳細は各 spec または実装フェーズで定義。」
- **依存**  
  - 資源（レア含む）→ Item マスタが存在すること。  
  - 設置 → FacilityInstance（または同等）が存在すること。  
  - 研究解放・設計図解放 → 研究 spec / 026 で「設備・型の解放」が定義されていること（後からでも可）。
- **実装順序の目安**  
  設備の「型」と「型ごとの建設レシピ」を仕様として固定し、設置・研究・探索は後から繋いでもよい。017 は「型と建設は資源消費」が決まっているので、DB 設計は実装タスクに含めればよい。

---

## 3. 懸念なしと判断できる点

- 用語（設備・型・基本型・派生型）は 017 と他 doc で統一済み。
- 数値設計（倍率式・配備キャラの基礎ステータス参照・三軸でバランス）は 017 に明記済み。
- 建設は型ごとの建設レシピで資源を消費（希少部品必須ではない）。レア資源の市場・スタック・ゲーム通貨のみは 017 で確定。
- 探索・市場の「詳細未定」は 017 内で明記済み。spec では「探索報酬・市場は別 spec に委ねる」と書けばよい。

---

## 4. まとめ

- **このまま spec 作成してよい。** 017 は設計 doc として完成しており、依存（Item, FacilityInstance, 型マスタ, 研究, 探索）は「実装時に必要」「別 spec で定義」と書いておけばブロッカーにはならない。
- spec では「017 設計（docs/017_facility_variant_and_rare_parts.md）に準拠する。Item・設置・型マスタ・研究・探索は各 spec または実装タスクで定義する」と一行入れておくと、あとで実装する人が迷わない。
