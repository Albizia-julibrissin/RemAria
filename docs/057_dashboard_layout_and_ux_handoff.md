# 057: ダッシュボードレイアウト案と UX アイデアの受け渡し方

ダッシュボードのレイアウト検討と、企画者↔実装（AI）間でレイアウト・UX 案を渡す際のフォーマットを定義する。

---

## 1. 前提（コンテンツ体験のコア）

- **ループの中心**: 探索（テーマ→エリア→出撃→戦闘/技能→報酬→帰還）
- **前提フロー**: 作戦（作戦室でプリセット・作戦スロットを設定）→ 探索でそのプリセットを選択
- **UX 目標**: このコア体験から逆算し、操作しやすい画面にする

---

## 2. 現状のダッシュボード構成（要約）

| ブロック | 内容 |
|----------|------|
| ヘッダー相当 | タイトル「ダッシュボード」＋ 所持通貨 |
| メニュー | `<details>` で開閉。宿舎・雇用・工業・研究・クラフト・倉庫・クエストへのリンク |
| メイン | 左: 探索（説明文・進行中サマリ・探索開始フォーム）。右: 作戦室カード・（管理者時）コンテンツ管理 |

- 探索が前面、作戦室は右サイドのカードで常時見える設計になっている。

---

## 3. レイアウト案（検討用）

### 案 A: 探索ファースト・メニューは脇

- **1 画面目で見せるもの**: 探索ブロックを最大にして、進行中探索サマリ or 探索開始フォームを中央に。
- **作戦室**: 探索ブロックの上 or 右に「作戦を編集」の入口を 1 つ。未設定時は「作戦を設定してください」と導線。
- **その他メニュー**: ヘッダーまたは左上のドロワー／「メニュー」開閉で宿舎・工業などへ。
- **利点**: 「やりたいこと＝探索」が一目で分かる。**欠点**: メニューが一覧しづらい場合あり。

### 案 B: ハブ型・カードグリッド

- **1 画面目**: 探索・作戦室・宿舎・工業・クラフト・倉庫・クエストなどを**同列のカード**で並べる。
- **探索カード**: 少し大きく or 上部に配置し、「今やること」として強調。
- **作戦室**: 探索の隣 or 直上に配置し、「探索の前にここ」と分かるようにする。
- **利点**: 全機能が一覧できる。**欠点**: 探索が「特別」になりにくく、初見で迷う可能性。

### 案 C: フロー導線型（ステップ表示）

- **1 画面目**: 「1. 作戦を決める → 2. 探索に出る → 3. 報酬を受け取る」のようにステップを短く表示。
- **ブロック**: ステップ 1 は作戦室へのリンク／要約、ステップ 2 が探索開始 UI、ステップ 3 が進行中 or 結果受け取り。
- **メニュー**: ステップの下 or 別エリアで宿舎・工業など。
- **利点**: 前提（作戦）→本編（探索）の順序が明示される。**欠点**: 慣れたユーザーには冗長に感じる場合あり。

### 案 D: 現状に近い・微調整

- 現状の「左: 探索 / 右: 作戦室＋管理」を維持しつつ、
  - 進行中探索があるときはそのブロックを最上部に
  - プリセット未設定時は探索開始フォームの上に「作戦室でプリセットを設定してください」を表示
  - メニューは `<details>` のまま or 常時表示の横並びリンクに変更
- **利点**: 変更が少ない。**欠点**: レイアウトの大胆な見直しはしない。

---

## 4. あなた→AI（企画者→実装）への渡し方

レイアウト・UX のイメージを渡すとき、次のいずれか（または組み合わせ）を使うと実装しやすい。

| 手段 | 内容 | 向いていること |
|------|------|----------------|
| **テキストのゾーン説明** | 「上から：タイトル、その下に探索ブロック、右に作戦室。メニューは左上のアイコンで開く」のように**ブロック単位で順序・位置**を書く | 構成の優先度・順序の指定 |
| **ワイヤー風 Markdown** | セクション見出しとリストで画面を上から書く（例: `## ヘッダー` → `## 探索` → `- 進行中サマリ`） | 情報の階層・何をどこに出すか |
| **ASCII レイアウト** | 四角や記号でざっくり配置を描く（下記サンプル参照） | 左右・上下の関係をざっくり伝える |
| **既存ページへの差分** | 「今の dashboard/page.tsx の ○○ を △△ に変えたい」とファイル＋変更意図を書く | 既存実装の微調整 |
| **参照 URL や画像** | 「このゲームのダッシュボードっぽく」など参考 URL やスクショを貼る | トーン・雰囲気の共有（実装は要約して文章化すると確実） |

**ASCII レイアウトのサンプル（案 A 風）:**

```
+------------------------------------------+
| ダッシュボード         [所持通貨]         |
+------------------------------------------+
| [≡ メニュー]  探索                       |
+------------------------------------------+
| +------------------------+  +----------+ |
| | 進行中の探索（あれば）   |  | 作戦室   | |
| | テーマ/エリア・続ける→  |  | プリセット| |
| +------------------------+  | 編集     | |
| | 探索を開始              |  +----------+ |
| | [テーマ][エリア][プリセット] [開始]   | |
| +------------------------+               |
+------------------------------------------+
```

---

## 5. AI→あなた（実装→企画者）への返し方

レイアウト案や変更案を返すとき、次の形式で書くようにする。

| 形式 | 内容 |
|------|------|
| **方針 1 行** | 「探索を最大ブロックにして、作戦室はその右に固定」など、何を優先したか。 |
| **ブロック構成** | 上から順に、どのブロックがどこにあり、何を表示するか。 |
| **変更点リスト** | 現状との差分を「○を△に変更」で箇条書き。 |
| **未決・選択肢** | 「メニューは開閉のままにするか、常時表示にするか」など、判断が必要な点。 |

必要なら、この doc の「案 A/B/C/D」や「ASCII サンプル」をコピーして差分だけ書き換え、チャットに貼ってもよい。

---

## 6. 次のステップ（例）

1. 上記の案 A〜D のどれに近いか（または組み合わせ）を決める。
2. 「4. あなた→AI」のいずれかの形式で、具体的なレイアウト案を 1 つ書く（または既存案を修正する）。
3. 実装側で `src/app/dashboard/page.tsx` をその案に沿って変更する。
4. 必要なら「5. AI→あなた」の形式で代替案やトレードオフを返し、合意したら doc の「採用案」として追記する。
