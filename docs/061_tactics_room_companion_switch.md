# 作戦室：仲間・メカ切り替え時の作戦・スキルが編集できない問題

作戦室で「仲間」または「メカ」のドロップダウンを別キャラに切り替えたとき、切り替え先の作戦スロットやスキル設定が正しく表示・編集できない事象の原因と対策案。

---

## 1. 事象

- 初期プリセット: 主人公・仲間１・メカ
- 仲間１の作戦を編集する
- 仲間枠を「仲間２」に切り替える
- **このとき仲間２の作戦にならず、スキル設定などもできない**

---

## 2. 原因

### 2.1 データの取得タイミング

- **page.tsx（Server Component）** は URL の `presetId` でプリセットを取得し、**その時点の編成 3 人**（slot1, slot2, slot3）の `characterIds` だけを元に以下を取得している。
  - `getTacticsForCharacters(characterIds)` … 3 人分の作戦スロット
  - `getBattleSkillsForCharacters(characterIds)` … 3 人分の習得スキル
  - `getTacticsSkillCatalogForCharacters(characterIds)` … 3 人分のスキルカタログ
- クライアント（TacticsEditorClient）には **初回レンダー時点の 3 人分のデータだけ** が `initialTactics` と `battleSkillsByCharacter` として渡る。

### 2.2 クライアント側の状態

- 仲間・メカのドロップダウンは **クライアントの state**（`slot2Id`, `slot3Id`）で制御されている。
- ユーザーが「仲間２」に切り替えると:
  - 表示上は「仲間２」の名前と作戦テーブルが表示される。
  - しかし **仲間２のデータはサーバーから渡っていない** ため:
    - **作戦スロット**: `slotsByCharacter` は初回の 3 人（主人公・仲間１・メカ）でしか初期化されていない。仲間２の key が無いため、フォールバックのデフォルト 1 行だけ表示され、**保存済みの仲間２の作戦は読み込まれない**。
    - **スキル選択**: `battleSkillsByCharacter[仲間２のid]` が存在しないため、行動の「スキル」プルダウンが空になり、**スキル設定ができない**。

まとめると「**編成をクライアント側で差し替えているが、差し替え先キャラの作戦・スキルデータを取得していない**」ことが原因。

---

## 3. 対策案

### 案A: 編成変更時に「編成だけ保存」してから再読み込み

- 仲間・メカのドロップダウンを変更したら、**編成（slot2/slot3 の characterId と列）だけ先に保存**し、保存成功後に `router.refresh()` でページを再取得する。
- 再取得時はサーバー上のプリセットがすでに「仲間２・メカ」になっているので、`getTacticsForCharacters` / `getBattleSkillsForCharacters` で仲間２・メカのデータが取れ、正しく表示・編集できる。
- **メリット**: 実装が単純。既存の Server Component の流れをそのまま使える。
- **デメリット**: 枠を変えるたびに保存＋再読み込みが走る。「編成だけ変えてまだ保存したくない」という使い方はできない。

### 案B: 編成変更時にクライアントで追加取得

- slot2 / slot3 の選択が変わったら、**増えた characterId についてだけ** Server Action で `getTacticsForCharacters` と `getBattleSkillsForCharacters` を呼ぶ。
- 取得結果をクライアントの state にマージする（`slotsByCharacter` に新キャラの作戦をセット、`battleSkillsByCharacter` に新キャラを追加）。スキルカタログも新しい 3 人で再取得するか、必要ならマージする。
- **メリット**: ページ遷移・リロードなしで、切り替え直後から作戦・スキルを編集できる。
- **デメリット**: クライアントの state 更新がやや複雑。新キャラ用のデータ取得ロジックと、既存 state とのマージを明示的に書く必要がある。

### 案C: 初回ロードで「全仲間・全メカ」の作戦・スキルを取得

- 作戦室の初回表示時に、**現在の編成 3 人に加え、仲間一覧・メカ一覧の全 characterId** に対して `getTacticsForCharacters` と `getBattleSkillsForCharacters` をまとめて呼ぶ（または 3 人＋候補全員を 1 回で渡す）。
- クライアントには「編成候補になり得る全員分」のデータを渡す。ドロップダウンで誰に切り替えても、すでにデータがあるのでそのまま表示・編集できる。
- **メリット**: 切り替えが一瞬。追加の非同期リクエストが不要。
- **デメリット**: 仲間・メカ数が多いと初回ロードが重くなる。作戦室は編成候補が十数人程度なら許容できる想定なら有効。

---

## 4. 推奨

- **短期対応**: **案B（編成変更時の追加取得）** を推奨する。  
  - 編成を「まだ保存しない」まま別キャラの作戦を確認・編集したいニーズに応えられる。  
  - 実装は「slot2Id / slot3Id の変更を検知 → 新規に含まれる characterId の tactics / battleSkills を取得 → state マージ」に整理すればよい。
- **案A** は「枠を変えたら即保存」でよい運用なら実装コストが最も小さい。
- **案C** は、仲間・メカの数が少ない（例: 10 人未満）前提なら、初回負荷と実装の単純さのバランスが良い。人数が増えると案B の方が安全。

---

## 5. 実装時の注意（案B を選ぶ場合）

- `TacticsEditorClient` 内で `slot2Id` / `slot3Id` の変更を `useEffect` で検知し、**現在表示対象の 3 人**のうち、`slotsByCharacter` または `battleSkillsByCharacter` に存在しない characterId があれば、その id に対して Server Action で取得してマージする。
- 既に存在するキャラ（例: 仲間２→仲間１に戻す）は再取得不要。
- 保存時は現状どおり「現在の slot1, slot2Id, slot3Id の 3 人分の `slotsByCharacter` を保存」でよい。
- spec/039 の「0.2 操作詳細」にある「キャラクター選択済みの枠にだけ作戦設定 UI を表示」は維持する。未選択枠は編集不可のままとする。

---

## 6. 実装状況

- **案B** を実装済み（spec/039 に 0.4 を追加、`TacticsEditorClient` で編成変更時に不足キャラ分を取得して state マージ）。
