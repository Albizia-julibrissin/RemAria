# 作戦スロット・戦闘処理のテーブルとアプリ仮案

本書は **作戦スロット（ブラックホール→ハイスピアのコンボをスロットに入れて実行する流れ）** を、どのテーブル設計とアプリ処理で実現するかの仮案。  
**主語・条件・状態異常などの拡張**を将来やりやすくするため、スキーマと処理の切り分けを慎重に考える。

前提: `14_battle_core_spec.md`, `14_battle_attributes_tactics.md`, `08_database_schema.md`。  
**スキルに必要そうな項目とシステム**の検討（草案ベース）は `14_battle_skill_design_draft.md` にまとめている。

---

## 1. 方針（念頭に置くこと）

- **主語の拡張**：今後「主人公枠」「仲間枠」「味方全員」などを増やす可能性がある。テーブルは「主語を識別子（コード）で持つ」形にし、**新しい主語はコードの追加＋アプリの評価ロジック追加**で済むようにする。
- **条件の拡張**：条件種別（hpBelowPercent, subjectHasAttrState など）も同様に**コードで持つ**。パラメータは「種別ごとに意味が違う」ので、**JSON または汎用カラム数本**で持ち、アプリで解釈する。こうすると条件種別を増やしてもスキーマ変更を最小にできる。
- **状態異常の拡張**：状態異常（デバフ）の種類は今後増える。**状態異常はマスタ（または enum 的テーブル）で管理**し、戦闘中の「付与されている状態」は実行時データ（メモリ上のバトルコンテキスト）で持つ。DB には「どの状態異常が存在するか」の定義だけを持つ。
- **テーブルとアプリの役割**：**何を「永続」し、何を「実行時だけ」持つか**をはっきりさせる。作戦スロット・スキル定義は永続。戦闘中の HP/MP/属性状態/状態異常は永続しない（戦闘結果だけ保存するかは別仕様）。

## 2. 想定フロー（ブラックホール → ハイスピア）

1. **戦闘開始前**  
   - ユーザーはパーティプリセット（主人公・仲間・メカ 3 体）を組み、**各キャラごとに作戦スロット 10 個**を設定済みとする。
   - 例：キャラ A のスロット 1「相手のいずれかが圧縮状態だったら → ハイスピア」、キャラ B のスロット 2「自分の MP が 20％以上だったら → ブラックホール」、… など。

2. **戦闘開始**  
   - プリセットから 3 体の Character を読み込み、**各キャラに紐づく 10 スロット**をそれぞれ取得する。敵 3 体・敵の位置も決定。  
   - 各キャラ・敵の「属性状態」を空で初期化。サイクル = 1。

3. **各サイクル**  
   - 生存者で行動順を抽選。  
   - **自分のターン**が来たキャラ A について：  
     - キャラ A に紐づく「作戦」は **そのキャラ用の 10 スロット**（作戦スロットはキャラごと）。  
     - 現在の戦闘コンテキスト（サイクル、本サイクル内の行動順、味方 3 体の HP/MP/状態異常、敵 3 体の HP/MP/属性状態、正面の相手など）を使って、スロット 1 から順に条件を評価。  
     - 最初に true になったスロットの「行動」を採用 → **通常攻撃** または **スキル ID**。  
     - スキルの場合、**発動時に MP を確認**し、不足していれば **不発**（スキルは発動せず。MP 消費なし・ダメージ・効果・属性状態付与なし）。そのターンは「不発」として行動済み扱いとし、次のスロットへはフォールバックしない。  
     - ターゲットは **列ウェイトで抽選**（誰を狙うかはスロットでは決めない）。  
     - 行動を実行（不発でない場合）：命中判定 → ダメージ計算（種別・属性耐性・直撃/致命）→ スキル効果（「対象が圧縮状態なら 1.5 倍＋圧縮消費」）を適用 → 命中時に対象に**攻撃の属性状態**を付与。  
     - 必要なら MP 消費・MP 回復。  
   - 全員行動したらサイクル終了。属性状態の「残りサイクル」を 1 減らす（または「付与サイクルを含めて 2 サイクル」のルールで消すタイミングを判定）。次のサイクルへ。

4. **ブラックホールを使ったターン**  
   - 誰かが「ブラックホール」を選択し、列ウェイトで選ばれた敵に命中 → その敵に **圧縮状態**（付与したサイクルを含めて 2 サイクル）を付与。

5. **そのあとハイスピアを使うターン**  
   - 別の味方（または同じ味方の 2 サイクル目）のターンで、スロット評価により「相手のいずれかが圧縮状態だったら → ハイスピア」が true になる。  
   - 行動はハイスピア。ターゲットは**また列ウェイトで抽選**なので、必ずしも圧縮状態の相手に当たらない（いやらしさ）。  
   - 当たった相手が圧縮状態なら、ダメージ 1.5 倍＋圧縮状態を消費。当たった相手には**穿孔状態**を付与。

この一連の流れを、**テーブル**と**アプリの処理**でどう分担するかを以下で整理する。

## 3. テーブル設計案

### 3.1 パーティプリセットと作戦の紐づけ

- **PartyPreset**（既存案を採用）  
  - userId, name（または presetIndex）, slot1CharacterId, slot2CharacterId, slot3CharacterId  
  - 1 ユーザーが複数プリセットを持てる想定。  
- **作戦スロットは「キャラごと」**に持つ。  
  - 各 Character が自分の 10 スロットを持つ。編成を変えても、キャラに紐づいた作戦はそのまま使える。  
  - テーブル: **TacticSlot**（下記）。

### 3.2 作戦スロット（TacticSlot）

| カラム | 型 | 説明 |
|--------|-----|------|
| id | String (cuid) | PK |
| characterId | String | FK → Character.id。このキャラの作戦として紐づく。 |
| orderIndex | Int | 1～10。評価順。 |
| subject | String | 主語コード。例: `self` / `any_ally` / `any_enemy` / `front_enemy`。**拡張時はコード追加＋アプリで解釈**。 |
| conditionKind | String | 条件種別。例: `hp_below_percent` / `hp_above_percent` / `subject_has_attr_state` / `cycle_even` / `turn_order_in_range` など。**拡張時はコード追加＋アプリで解釈**。 |
| conditionParam | Json? | 条件に依存するパラメータ。例: `{ "percent": 40 }`, `{ "attr": "crush" }`, `{ "turnMin": 1, "turnMax": 2 }`, `{ "cycleModulo": 3 }`。アプリが conditionKind に応じて読む。**新条件追加時はスキーマ変更不要**。 |
| actionType | String | `normal_attack` / `skill`。 |
| skillId | String? | actionType = skill のとき FK → Skill.id。 |

- **主語・条件種別はコードで持つ**ことで、将来「主語を 1 つ増やす」「条件を 1 つ増やす」は、**マスタや定数＋アプリの評価関数の追加**で済む。DB のカラムは増やさない。
- conditionParam は JSON 推奨。種別ごとにキーが違うので、スキーマを変えずに拡張できる。

### 3.3 作戦設定画面（UI 想定）

- 作戦設定画面では **キャラを 3 人選び**、**3 人分のスロットを同一画面に並べて表示・編集**できる UI を想定する。
- 1 画面でパーティ全体の作戦（各キャラの 10 スロット）を確認・設定できる形にし、編成とコンボの見通しを立てやすくする。

### 3.4 スキル（Skill）の戦闘用拡張

戦闘で使うスキルには次の情報が必要。

- **攻撃種別**：物理 / 魔法（PATK/PDEF か MATK/MDEF か）。
- **属性**：無・圧縮・切創・穿孔・焼損・凍傷・侵食・極性（命中時に付与する属性状態）。
- **条件付き効果**：「対象が〇〇状態のとき、ダメージ倍率・その状態を消費」など。

**拡張を考えた設計**：

- **Skill に戦闘用カラムを追加**（最小限で固定概念のみ）  
  - `battleAttackType` String? … `physical` / `magic`。NULL なら戦闘スキルでない。  
  - `battleAttribute` String? … `none` / `crush` / `slash` / `pierce` / `burn` / `freeze` / `corrode` / `polarity`。命中時に付与する属性状態。  
  - **戦闘ログ用メッセージ**（下記）。  
  - これらは「攻撃の性質」なので、スキル種類が増えても列挙で足りる想定。

#### 戦闘ログ用メッセージ（演出テキスト）

- スキルごとに **通常メッセージ** と **条件達成時の追加メッセージ** を持たせ、戦闘ログに表示する。  
- **通常メッセージ**（例：`全てを飲み込む崩壊の暗闇が{敵名}を飲み込む・・・！`）  
  - プレースホルダー：`{敵名}` / `{攻撃者名}` など、表示時に実際の名前で置換。  
  - スキル発動時（不発でない場合）に 1 行目として表示。  
- **条件達成時の追加メッセージ**  
  - そのスキルの **SkillBattleEffect** が発動したとき（例：対象が圧縮状態で 1.5 倍＋消費したとき）に、通常メッセージの後に追記して表示する。  
  - UI では **文字色を変えて** 条件達成部分であることが分かるようにする。  
- テーブル案：Skill に `battleLogMessage`（通常文）、`battleLogMessageOnCondition`（条件達成時追加文、NULL 可）を追加。  
- 不発時は「〇〇は MP が足りず、スキルは発動しなかった」などの専用ログを出してもよい。
- **条件付き効果**は種類が増えうるので、**別テーブル SkillBattleEffect** で持つ。  
  - 1 スキルに 0 個または複数（例：ハイスピアは「対象が圧縮なら 1.5 倍＋圧縮消費」の 1 個）。  
  - 将来「凍傷状態なら追加ダメージ」なども行を足すだけで表現できる。

**SkillBattleEffect 案**

| カラム | 型 | 説明 |
|--------|-----|------|
| id | String (cuid) | PK |
| skillId | String | FK → Skill.id |
| triggerAttrState | String | 対象がこの属性状態を持っていたときに発動。例: `crush`。 |
| damageMultiplier | Decimal? | 発動時のダメージ倍率。例: 1.5。 |
| consumeAttrState | Boolean | 発動時にその属性状態を消費するか。 |

- トリガーが「状態異常（デバフ）」の場合は、将来 `triggerType` を `attr_state` / `debuff` のように分け、`triggerDebuffCode` を足すなどして拡張できる。**今回は属性状態のみ**とする。

### 3.5 状態異常（デバフ）の定義

- **主語が状態異常**（`selfHasDebuff`）を評価するには、「どの ID が状態異常か」の定義が必要。  
- 案：**Debuff マスタ**（または StatusEffect のような名前）を 1 テーブル用意し、id / code / name などを格納。戦闘中「このキャラが持っている状態異常」は、実行時には id または code のリストで持つ。  
- 状態異常の種類を増やすときは、**このマスタに行を足す＋アプリで付与/解除ロジックを足す**だけで済む。  
- MVP で状態異常をまだ使わない場合は、テーブルだけ用意して「作戦条件では未使用」にしてもよい。

### 3.6 属性・主語・条件のコード管理

- **enum をアプリで定義**し、DB に保存するときはその文字列（コード）を入れる。  
- あるいは **Attribute / TacticSubject / TacticConditionKind のような小さなマスタ**を用意し、一覧は DB から読む。どちらでも「追加はコード or 行の追加」で済む。  
- 本仮案では、**アプリ側の定数（enum）と DB の文字列を一致させる**形を推奨。マスタ化は「管理画面で並び順を変えたい」などが必要になった段階で検討。

## 4. アプリ処理の分担

### 4.1 戦闘開始時

- 出撃する 3 体の Character ごとに、その characterId に紐づく **TacticSlot** を orderIndex 昇順で取得（キャラあたり最大 10 件）。  
- 3 体の Character と習得スキル（CharacterSkill → Skill）を取得。  
- 各 Skill の **SkillBattleEffect** を取得し、スキル ID → 効果リストのマップを作る。  
- 戦闘用の**コンテキスト（メモリ上）**を組み立てる：  
  - 味方 3 体・敵 3 体の、現在 HP/MP、位置（row/col）、**属性状態のリスト**（例: `[{ attr: 'crush', expiresAfterCycle: 2 }]`）、状態異常（デバフ）のリスト。  
  - 現在サイクル、本サイクル行動順リスト。

### 4.2 自分のターン：作戦評価

- そのキャラの**スキルごとの残りクール**を、ターン開始時に 1 減算（0 未満にしない）。その後、作戦評価を行う。
- 現在のコンテキストと**CT 情報**（このキャラで「残りクール > 0」のスキル ID 一覧）を使って、**条件評価関数**を 1 スロットずつ呼ぶ。
  - 入力：subject, conditionKind, conditionParam、コンテキスト（味方/敵の HP・MP・属性状態・状態異常、サイクル、行動順インデックス、正面の相手など）、および「このスキルは CT 中か」の判定。
  - 出力：boolean。
- **条件を満たした**スロットの行動がスキルの場合、そのスキルが**CT 中**ならそのスロットは採用せず、**次のスロットを評価**する（フォールバック）。詳細は `docs/023_skill_cooldown_ct_design.md`。
- 評価は**アプリのロジック**に集約する。
  - 例：`subject_has_attr_state` + `{ "attr": "crush" }` なら、「subject が any_enemy のとき、敵 3 体のうち誰か 1 人でも crush 状態を持っていれば true」。
  - 主語・条件種別の追加は、この**評価関数の分岐追加**と **conditionParam の仕様**を doc に書くだけで対応。

### 4.3 行動実行とスキル効果

- 行動が **skill** の場合、Skill の battleAttackType / battleAttribute でダメージ種別と付与する属性状態を決定。  
- ダメージ計算後、**SkillBattleEffect** を参照：対象が `triggerAttrState` の状態を持っていれば、`damageMultiplier` を掛け、`consumeAttrState` ならその状態を削除。  
- その後、命中時に対象に **attack の battleAttribute** を付与（付与サイクルを含めて 2 サイクル）。

### 4.4 ターゲット抽選

- 作戦で「誰を狙うか」は決めない。行動が決まったあと、**列ウェイト**で敵 1 体を抽選。既存の `pickEnemyTarget` と同様のロジックを、味方 3 体が攻撃する場合にも使う。

## 5. まとめと拡張時のチェックリスト

- **主語を増やす**  
  - テーブル：変更なし（subject はコード文字列のまま）。  
  - アプリ：条件評価で subject の分岐を 1 つ追加。  
  - doc：`14_battle_attributes_tactics.md` の主語一覧を更新。

- **条件種別を増やす**  
  - テーブル：変更なし（conditionKind はコード、conditionParam は JSON）。  
  - アプリ：条件評価で conditionKind の分岐を 1 つ追加。conditionParam のキーを定義。  
  - doc：条件一覧に追加。

- **状態異常を増やす**  
  - テーブル：Debuff マスタに行追加（またはアプリの enum に追加）。  
  - アプリ：付与/解除のロジック、および `selfHasDebuff` の判定に新しい状態異常を含める。  
  - 戦闘中の「誰がどの状態異常を持っているか」は永続しない（実行時コンテキストのみ）。

- **スキルの条件付き効果を増やす**  
  - テーブル：SkillBattleEffect に列を足すか、triggerType で「状態異常トリガー」を増やす。  
  - アプリ：行動解決時に Effect を適用する箇所で、新しい triggerType を処理。

この仮案で、**ブラックホール→ハイスピアを作戦スロットに入れて実行する流れ**と、**主語・条件・状態異常をあとから増やしやすい形**を両立させている。実装時には、PartyPreset / TacticSlot / Skill 拡張 / SkillBattleEffect を Prisma に追加し、戦闘ループ側で「作戦評価 → 行動決定 → ターゲット抽選 → 行動解決（属性状態付与・消費）」をこの設計に沿って実装すればよい。
