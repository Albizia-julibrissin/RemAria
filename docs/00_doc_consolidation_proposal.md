# ドキュメント統廃合の提案

ドキュメントが `docs/` と `spec/` に分散してきたため、**統廃合の方向性**と**優先してやるとよいこと**をまとめる。  
実施するか・どこまでやるかは判断しやすいように「効果」と「手間」を併記する。

---

## 1. 現状の課題（簡潔に）

| 課題 | 具体例 |
|------|--------|
| **戦闘・スキルが多ファイル** | 14_battle_*.md が 4 本、14_初期スキル系、023, 041, 042, skill_fields_ja。spec/038～040 と合わせると 10 本以上。 |
| **番号体系がバラバラ** | 00～16 のあと 017, 018… と 3 桁。020～023。041, 042。14 だけサブが複数。 |
| **「どれが正本か」が分かりにくい** | 戦闘の「正」は spec/038 だが、詳細は 14_battle_core_spec や 042 に書いてある。 |
| **参照のたびに複数ファイルを開く** | 効果を 1 つ追加するだけで 042 / 038 / 041 / seed を触る可能性。 |

---

## 2. 統廃合の基本方針（おすすめ）

- **全部 1 本にまとめる必要はない**。テーマごとに「入口 1 本 + 必要ならサブ」にすると探しやすい。
- **spec は「契約」として残す**。設計メモや経緯は docs に寄せる。
- **索引を 1 か所に置く**。→ すでに `docs/README.md` で実施。

---

## 3. 具体的な統廃合案

### A. 戦闘・スキルまわり（効果が大きい）

**現状**: 14_battle_core_spec, 14_battle_attributes_tactics, 14_battle_skill_design_draft, 14_battle_tactics_impl_draft, 14_初期スキル_評価と新規効果, 023, 041, 042, skill_fields_ja, spec/038…

**案 1（おすすめ）: 統合はしない。索引で整理して「入口」を決める**

- `docs/README.md` で「戦闘・スキル」を 1 ブロックにまとめ、読む順序を書いてある（済）。
- **正本の役割をはっきりさせる**:
  - **仕様（契約）**: spec/038（API・Skill/SkillEffect・ルール）。変更時はここを更新。
  - **効果一覧・param**: 042。新規 effectType はここに追記。
  - **実装の流れ・Phase**: 041。Phase が進んだら 041 を更新。
- 14_battle_skill_design_draft は「過去の設計メモ」。参照頻度が低ければ「アーカイブ」フォルダに移すか、先頭に「設計メモ（正本は spec/038 と 042）」と注釈を足すだけでもよい。

**案 2: 14_* を 1 本にマージする（手間大）**

- `14_battle_core_spec.md` を土台に、14_battle_attributes_tactics / 14_battle_skill_design_draft / 14_battle_tactics_impl_draft のうち「まだ参照する部分」をセクションとして取り込む。
- ファイル数は減るが、1 ファイルが長くなり、差分レビューがしづらくなる。**今の段階では推奨しない**。

**案 3: 042 と 041 の役割をさらにはっきりさせる**

- 042 = 「effectType と param のリファレンスだけ」に限定（すでにそうなっている）。
- 041 = 「実装 Phase・データフロー・型」に限定。effectType の説明は「042 参照」に寄せる。
- 重複を減らし、「効果の意味は 042」「実装の順序は 041」と分けられる。

---

### B. 番号体系をそろえる（任意）

**現状**: 00～16, 017～019, 020～023, 041～042。14 はプレフィックスで 4 本。

**案**: いったん変えなくてよい

- 番号を振り直すと、他ドキュメントや spec からの「docs/014_xxx」などの参照が一斉に壊れる。
- 代わりに **docs/README.md の索引**で「戦闘は 14_* と 023, 041, 042」とまとめてあれば、番号が不揃いでもたどれる。
- 新規ドキュメントを足すときだけ、「テーマごとのブロック」に従って命名する（例: 戦闘関連なら 14_ か 04x_）とよい。

## 4. 優先してやるとよいこと（まとめ）

1. **索引を活用する**  
   `docs/README.md` を「docs の入口」にし、リンク切れがないかだけたまに確認する。

2. **正本を決めて書く**  
   - 戦闘スキル仕様 → spec/038  
   - effectType の意味・param → 042  
   と決め、追加・変更時はそこに書く。他は「〇〇参照」で済ませる。

3. **設計メモは「メモ」と明示する**  
   14_battle_skill_design_draft などは「設計時のメモ。正本は spec/038 と 042」と先頭に 1 行書く。  
   必要なら `docs/archive/` のようなフォルダに移す。

4. **大きな統合はしない**  
   複数ファイルを 1 本にまとめると、差分が読みづらくなる。  
   「探しやすくする＝索引 + 正本の明確化」で十分。

---

## 5. やらなくてよいこと

- 全 doc を 1 ファイルにまとめる。
- 番号の全面振り直し（参照が壊れる）。
- spec と docs の境界をなくす（spec は契約として短く保った方がよい）。

---

**結論**: 統廃合は「**索引の整備（README）**」と「**正本の固定（038 + 042）**」「**古い設計メモの注釈 or アーカイブ**」までで十分。ファイルの大量マージや番号の付け直しはしなくてよい、という案。

---

## 6. Cursor（AI）の判断に効く是正

コーディングを Cursor に任せる前提なら、**「どのファイルが正本か」「タスクごとに何を読むか」** が AI に伝わるようにしておくと、判断ブレが減る。

### 6.1 起きがちな問題

| 問題 | 影響 |
|------|------|
| **正本が複数ある** | effectType の定義が spec/038 の表と docs/042 の両方にある。AI が 038 だけ読んで古い・短い方で実装してしまう。 |
| **「次に読むファイル」が書いていない** | 038 に「effectType で分岐」とだけあり、一覧が 042 にあると書いていない。AI が 042 を開かずにコードだけから推測する。 |
| **タスクとファイルの対応が暗黙** | 「新規 effectType を追加」するとき、038 / 042 / 041 / seed / run-battle のどれをどう更新するか、文書に手順がない。 |
| **古い状態が残る** | 041 の「Phase 8 未実装」のまま実装済みだと、AI が二重に実装したり逆に「まだ」と判断したりする。 |

### 6.2 是正のポイント（やるとよいこと）

1. **正本を明示する**  
   - spec/038 の effectType の節に「**一覧・各 param の定義は `docs/042_battle_effect_types_reference.md` を正本とする。** 本 spec は API と Skill/SkillEffect の構造の契約のみ。」と 1 行入れる。  
   - AI が 038 を読んだときに「詳細は 042」と分かる。

2. **正本側から「やること」を書く**  
   - 042 の冒頭に「**新規 effectType 追加時**: (1) このファイルに定義ブロックを追加 (2) `src/lib/battle/run-battle-with-party.ts` の effectType 分岐にハンドラを追加 (3) 必要なら `prisma/seed.ts` の BATTLE_SKILL_EFFECTS に追加。」と手順を 1 行で書く。  
   - タスクと参照ファイルが対応する。

3. **相互参照を入れる**  
   - 038 → 042 は上記。042 → 038 は「仕様の土台: spec/038」で済んでいる。  
   - 041 の「参照ドキュメント」に 042 を入れてあるので、実装 Phase を読む AI が 042 にたどり着ける。

4. **Cursor 用のルールを 1 枚置く（任意）**  
   - `.cursor/rules` やプロジェクト直下の `AGENTS.md` に「戦闘スキル効果（effectType）の定義・param は `docs/042_battle_effect_types_reference.md` を正本として参照する。」と書いておく。  
   - チャットで「effectType を追加して」と言ったときに、どのファイルを見るかが揃う。

5. **ステータスを更新する**  
   - 041 の Phase 表や「未実装」一覧は、実装が終わったら「実装済み」に更新する。  
   - AI が「まだだから実装する」と重複したり、「実装済みだから触らない」と誤解したりするのを防ぐ。

### 6.3 まとめ（Cursor 向け）

- **人間が読みやすくする**より、**「正本はこれ」「このタスクではこの順でこれを見る」** を文書と 1 行注釈ではっきりさせる方が、AI の判断に効く。
- 上記 1～3 と 5 は、既存ドキュメントに数行足すだけでできる。4 はルールファイルを 1 つ作るだけ。
- ファイルを減らすより、**参照の向きと正本の明示**をそろえることを優先するとよい。

## 7. Cursor 向け環境整備のまとめ（effectType 以外）

effectType 以外でも、実装時に Cursor が迷わないように次を用意している（または推奨）。

| 項目 | 場所 | 内容 |
|------|------|------|
| **実装時の参照一覧** | プロジェクト直下 **`AGENTS.md`** | spec↔実装の対応表、正本ドキュメント（042 等）、データモデル（prisma/schema）、コマンド（dev, seed, lint）、戦闘計算の参照先。 |
| **ドキュメント索引** | docs/README.md | テーマ別の doc 一覧。冒頭で AGENTS.md を案内。 |
| **アーキテクチャ・ディレクトリ** | docs/03_architecture_spec.md | 技術スタック、依存方向、命名規約。Cursor 運用ルールの最後に「AGENTS.md を参照」を追記済み。 |
| **正本の明示（effectType）** | spec/038, docs/042 | 038 に「effectType の詳細は 042 が正本」、042 に「新規追加時の手順」を記載済み。 |

今後の追加候補（必要になったらでよい）:

- **テストの実行方法**: テストを導入したら AGENTS.md の「コマンド・検証」に `npm test` 等を追記。
- **他ドメインの正本**: 戦闘以外（例: 設備タグ・工業スキル）で「この一覧はこのファイルが正本」と決めたら、038→042 と同様に相互参照を 1 行ずつ入れる。
- **.cursor/rules**: プロジェクト固有のルールを増やしたい場合は `.cursor/rules` に「AGENTS.md を参照」等の短いルールを置く。
