# ドメインモデル（暫定・工業フォーカス強化版）

この文書は、ゲーム内の主要概念（エンティティ）と関係、設計原則を定義する。
DB設計（Prisma）およびSpec作成時の前提とする。

---

## 主要エンティティ

### アカウント/キャラ/メカ（戦闘単位の統一）

- User：ユーザーアカウント
- **Character**：**主人公・仲間・メカを1テーブルで統一**。判別用に **category**（protagonist / companion / mech）を持つ。戦闘ではステータス・スキルを同じように扱う。
  - **category = protagonist**：アカウントキャラ（本人）。1ユーザーあたり必ず1体。User.protagonistCharacterId で参照。
  - **category = companion**：仲間。0〜N体。厳選要素の中心。
  - **category = mech**：追従メカ。0〜N体。パーツで性能/スキルが変化する想定（MechPart は別テーブルでパーツ構成を保持する場合あり）。
- 主人公専用の効果は Character にカラムを追加し、仲間・メカの行では NULL／未使用とする。
- MechPart：メカパーツ（個体。CAP値やサブステを持つ想定）。category = mech の Character に紐づける。

### アイテム
- Item：アイテム総称
  - Material：素材（鉄など）
  - Equipment：装備（武器/防具。生産で基礎値が上がる安定枠）
  - Relic：遺物（探索中心のハクスラ枠。4枠）
  - Blueprint：設計図（将来、設備/レシピ/パーツの解放に利用）

### 工業/設備
- Factory（または Workshop）：プレイヤーの工業管理単位（設備枠などの母体）
- FacilityInstance：設置された設備（プレイヤーが組み替え可能な設備群）
- FacilityType：設備（川探索拠点・精錬機などの施設の種類。採掘/精錬/製造/訓練など。型は docs/017）
- Recipe：入力→出力の定義（MVPでは「1設備＝1レシピ」を基本）
- Assignment：設備への配備（キャラ/メカを割り当てる）

### 研究（解放）
- ResearchLab：固定設備（別メニュー扱い）
- Tech：研究項目（解放条件と解放結果を持つ）
- Unlock：解放結果（設備/レシピ/Tierなどを使用可能にする）

### 非同期処理
- Job：時間経過で進行する処理（訓練/製造/採掘など）
- JobProgress（任意）：進行の表示補助（原則は startAt/finishAt で計算）

### 探索
- Expedition：探索（出撃・結果）
- Battle：戦闘（入力→ログ→結果）

## パーティと所有関係（概略）

User
 └ Character（category で判別：protagonist 1体 ＋ companion N体 ＋ mech N体）
    │  ※ アカウントキャラは User.protagonistCharacterId で特定。戦闘ではステータス・スキルを同一扱い。
    │  ※ メカはパーツ構成を MechPart 等で持つ想定（将来）
    ├ Inventory（Items）
    ├ Factory（1 または N）
    │   ├ FacilityInstances（N：設置枠内）
    │   └ FixedFacilities（研究室/倉庫：別メニュー）
    ├ Jobs（N：訓練/製造/採掘）
    └ Expeditions（N） → Battle（N）

## 固定設備と可変設備

### 固定設備（別メニュー / 常設）
- ResearchLab（研究室）：設備解放条件の確認・達成・アンロック
- Storage（倉庫）：容量・保管・在庫の閲覧/整理（設備枠に含めない）

狙い：
- 設置パズルの負担を減らす
- 初心者が詰まない
- 工業の導線を明確にする

### 可変設備（設置枠に置くもの）
- 採掘（資源探索）
- 精錬
- 加工（装備/パーツ製造）
- 訓練（ステ変動）

---

## 設備とレシピ（MVP設計原則）

### 原則：1設備＝1レシピ（MVP）
- FacilityInstance は基本的に1つの Recipe を持つ（固定）。
- 1レシピは「入力素材 → 出力物」「所要時間」「必要条件」だけを持つ。
- 複雑な多機能設備（複数レシピ切替）は後期拡張で扱う。

例：
- 鉄鉱山（採掘設備）：inputなし → 鉄
- 鉄精錬機：鉄 → 純鉄
- メカ頭パーツ機 Tier1：純鉄 → 頭パーツ（CAP50個体）
- 訓練所：inputなし → ステ変動（Job成果）

## 資源探索（エリア/ノードの考え方）

将来的に複数エリア・複数資源を扱うため、以下の概念を想定する：

- Region / Area（探索エリア）
- ResourceNode（資源ノード：このエリアで採れる資源と出力）

MVPでは簡略化してもよい：
- エリアごとに採掘できる資源を固定し、採掘設備がそれを出力する。

## 配備（Assignment）と工業効率

### 基本方針
- 設備は配備なしでも動作する。
- 配備すると工業効率が上がる（主に「製造時間短縮」）。
- “増産（歩留まり）”は経済破綻リスクがあるため、MVPは時間短縮中心。

### 配備先の考え方
- FacilityInstance に assignmentSlots を持つ（0〜n）。
- キャラ/メカを割り当てると、Recipeの所要時間に係数がかかる。

備考：
- アカウントキャラ（本人）は「研究室に常駐している」扱いとし、データ上のAssignmentは持たない（MVP方針）。

## スキル体系（戦闘/工業/才能）

戦闘では **Character を同一テーブルで参照**するため、ステータス・スキルの参照は**主人公・仲間・メカとも同じ扱い**。種別は **category**（protagonist / companion / mech）で判定する。

### アカウントキャラ（本人＝protagonist）
- 戦闘アクティブスキル（複数）
- 戦闘パッシブスキル（複数）
- 工業パッシブは原則持たない（差別化）
  - 将来、アカウント全体バフ等を入れる場合は「本人の工業スキル」として扱う。
- **主人公専用効果**：Character にカラムを追加（例: protagonistTalentId）。仲間では NULL／未使用。

### 仲間キャラ（Minion）
- 戦闘アクティブスキル（取得/育成で狙える想定）
- 戦闘パッシブスキル（取得/育成で狙える想定）
- 工業パッシブスキル（厳選要素）
  - 入手時にランダムで1つ所持
  - CAPが一定に達するごとに追加でランダム公開/獲得
  - 最大数は暫定で4程度を想定
  - 例：資源探索完了時、鉄の生産量 +3%

### 才能パッシブ（特別枠）
- ステータス合計が上限（CAP到達）に達した際、特別な才能パッシブを獲得する想定。
- ユニークに近い効果で、厳選要素の核となる。
  - 例：根性（CAPを超えてSTRに +20 補正）
- 才能パッシブと工業パッシブを、仲間厳選の主軸に置く。

## アイコン/キャラ拡張（将来）

- 仲間キャラは厳選要素として、アイコン/外見の多様性を重視する。
- 将来的に「自由投稿/公募でアイコンが増える」仕組みを想定。
- プレイヤーが好みのキャラを引き当てるまで回す、長期コンテンツにする。

---

## 非同期処理（Job）の位置づけ

Jobは以下を統一的に扱う：
- 訓練（ステ変動）
- 製造（レシピに従った生成）
- 採掘（素材生成）

Jobは原則、startAt/finishAt を保存し、進行は計算で扱う。
完了時に成果（生成物/ステ変動/消耗）を確定し、DBに反映する。

---

## 設計原則まとめ

- 戦闘ロジックは純粋関数として扱える形（入力→結果）にする。
- 固定設備（研究室/倉庫）と可変設備（設置枠）を分離する。
- MVPは「1設備＝1レシピ」でシンプルにする。
- 配備は任意、MVPは時間短縮中心。
- 仲間は「工業パッシブ」と「才能パッシブ」を厳選の核にする。
