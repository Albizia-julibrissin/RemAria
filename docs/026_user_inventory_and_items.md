# 026: ユーザーアイテム所持仕様（設計メモ）

ユーザーの「持ち物」全体を定義し、資源・装備・メカパーツ・遺物・スキル分析書・設計図・課金アイテムの扱いを整理する。  
**docs/021（装備サイクル）・02（ドメイン）・09（装備枠）・019（在庫）・017（希少部品）・044（メカパーツ）** と接続する。

---

## 1. 製造の役割分担（重要）

**装備・メカパーツ・探索用消耗品は工業設備では作らない。**

- **工業設備**：資源の**生成**（資源探索）と**加工**（精製など）のみ。完成品の装備・メカパーツは作らない。
- **アイテムクラフト**：ユーザーが手動で開く**別メニュー**。生産資源・材料を消費して、**装備・メカパーツ・探索用消耗品**を作る。

理由：1 レシピ 1 設備にすると装備種別ごとに設備が必要になり、設備の入れ替えが激しくなるため。工業設備は「材料を作る」専用とし、完成品クラフトはクラフト専用メニューに集約する。

```
工業設備 → 装備・メカパーツ・消耗品の「材料」を作る
アイテムクラフトメニュー → 資源・材料を消費して装備・メカパーツ・消耗品を作る
```

## 2. 種別ごとの仕様（想定内容の整理）

### 2.1 資源

- **取得元**
  - 工業設備の生産（資源探索・精製機などで取得）
  - 加工した資源（精製機の出力など）
  - 探索ドロップのレア資源
- **所持**：スタック型（同一アイテムは数量で管理）。現在の **UserInventory（userId, itemId, quantity）** で扱える。
- **消費**：アイテムクラフトの入力、設備建設時の資源消費（017。消費資源に希少部品が含まれる型はあるが必須ではない）、その他レシピ入力。
- **補足**：レア資源は Item マスタの category やタグで「レア」と区別し、装備/メカパーツのクラフト入力に使う想定（021・017）。

### 2.2 装備

- **製造**：**アイテムクラフトメニュー**で、生産資源・材料（＋必要に応じてレア資源）を消費して製造。工業設備では作らない。
- **性能**：製造時にステータスを**種別ごとの範囲内でランダム**に決定。**個体差はステのみ**（同一種別でもステ配分が個体ごとに異なる）。補正対象は**基礎ステから戦闘用ステに解釈した後の値**（物理攻撃力など）。**属性耐性**の補正もある（種別・個体で定義）。
- **装着**：主人公・仲間のみ。装備枠は **6 種 ＋ 遺物 4 枠**（下記）。
- **装備枠（6 種）**：主武器・副武器・頭・胴・腕・脚。MVP ではそのうち **2 種程度**を先行実装する想定（例：鉄シリーズ＝武器系、布シリーズ＝防具系など）。
- **装備メニュー**：一覧・着脱・比較表示など（別途 UI 検討）。

→ データ上は「装備マスタ（EquipmentType 相当）」と「装備個体（EquipmentInstance）」の2層。User は複数個体を所持し、そのうちの1つをキャラのスロットに装着する形。

### 2.3 メカパーツ

- **製造**：**アイテムクラフトメニュー**で、生産資源・材料を消費して製造。工業設備では作らない。仕様は装備と同様に「材料は工業、完成品はクラフト」。
- **性能**：製造時にステータスを範囲内でランダムに決定。**個体差はステのみ**。補正対象は**基礎ステータス**（024・044）。**属性耐性**もある（現状未実装のため、仕様策定後に実装）。
- **装着**：メカのみ。部位は **spec/044 の 6 スロット**（frame / core / head / rightArm / leftArm / legs）。メカも**遺物 4 枠**を装備可能。既存 **MechaEquipment** を**個体（MechaPartInstance）**参照に拡張する形を想定。

→ 装備と同様に「パーツ種別マスタ（MechaPartType）」「パーツ個体（MechaPartInstance）」の2層。User は複数個体を所持し、メカの各スロットに1個体ずつ装着。

### 2.4 探索用消耗品

- **製造**：**アイテムクラフトメニュー**で、生産資源・材料を消費して製造。
- **消費**：探索中に使用して消費。効果は種別ごとに定義（回復・バフ・ダメージなど）。
- **所持**：スタック型（数量で管理）。

→ 資源と同様に Item マスタ＋UserInventory で扱うか、消費時効果の種類に応じて category 等で区別する。

### 2.5 遺物

- **入手**：探索ドロップのみ（現時点想定）。
- **性能**：ドロップ時に**テーブル別でランダム**に決定。**個体別ユニーク**。
  - **特殊効果**：遺物専用の**パッシブ効果**を定義する（既存スキルIDの流用ではなく、遺物用効果テーブル）。例：「燃焼状態の相手への攻撃時ダメージアップ」など。
  - **どの型がどのエリアで出るか**：探索エリアごとにドロップテーブルを持つ。
  - **基礎ステ補正**：2つ持つ。1つ目はどのステかランダム、2つ目は1つ目と被らないようにする。**％補正**とする。
- **装着**：主人公・仲間・メカの各キャラ **4 枠**（09・021 と一致）。装備枠 6 種とは別に、遺物専用 4 スロット。
- **データ**：遺物型マスタ・遺物用パッシブ効果テーブル・エリア別ドロップテーブル・個体テーブル（RelicInstance）が必要。個体には「パッシブ効果ID」「基礎ステ補正1（stat + ％）」「基礎ステ補正2（別 stat + ％）」などを保持。

### 2.6 スキル分析書

- **入手**：探索ドロップ。
- **消費効果**
  - **初回使用**：そのスキルを習得し、レベル 0 にする。
  - **レベルアップ**：レベル 1 になるのに 1 冊、レベル 2 になるのに 2 冊（追加）、… の形で必要冊数を消費。
- **レベル効果**：レベル上昇でスキル倍率補正・MP 消費軽減。**sqrt(L/99)** で効かせる。数値は **spec/052 §4.3** に定義（物理・魔法：倍率 1.00→1.25、消費MP 1.0→0.9倍；補助：消費MP 1.0→0.8倍）。
- **所持**：スタック型（同一スキル用の分析書は数量で管理）。消費時に「習得済みか」「現在レベル」「次レベルに必要な冊数」を参照する。

→ スキルマスタ（Skill）とキャラの習得（CharacterSkill）に **level** を追加し、「次レベルに必要な分析書数」をレベルから算出するルール（例：Lv0→1 は 1 冊、Lv1→2 は 2 冊）を仕様で固定する形が考えられる。

### 2.7 設計図

- **入手**：探索ドロップなど（テーマ目標の「派生型設計図」など、020 で言及）。
- **消費**：**研究メニュー**で消費する。メニュー構成が「探索・研究・クラフト」である場合、**研究**のところで設計図を消費して各種「解放」を行う。
- **解放の種類**（いずれも研究での消費）：
  - **設備解放**：その設備種別を建て可能にする。
  - **設備型の解放**：その設備の派生型などを選択可能にする。建設時はその型の建設レシピで定義した資源を消費する（017。希少部品は必須ではない）。
  - **クラフトレシピ解放**：そのクラフトレシピをアイテムクラフトで利用可能にする。
- **所持**：**設計図と同じ持ち方**で統一する。クラフトレシピの解放用アイテムも、設計図と同様にスタック型で所持し、研究で消費して解放する。データ上は Item（category=blueprint 等）＋UserInventory で扱える。
- **補足**：02 ドメインの Blueprint（設備/レシピ/パーツの解放に利用）と一致。解放対象ごとに別アイテム（鉄の剣の設計図、〇〇設備派生型の設計図など）を用意し、消費時に「どの設備／どの型／どのクラフトレシピが解放されるか」をマスタで紐づける。

### 2.8 課金アイテム（今後の想定）

- **雇用権利書**：現在は User.companionHireCount で「回数」を直接付与しているが、最終的には「雇用権利書」をアイテムで所持し、**仲間作成画面で消費**することで仲間作成可能にする形を想定。
- **設備加速券**：**設備画面などで消費**し、生産完了を即時または短縮する。
- **その他**：随時追加。**消費は各機能の画面で行う**（雇用は雇用画面、加速は設備画面など）。**バッグ画面では所持一覧の確認のみ**でよい。

## 4. スコープ策定

### 4.1 MVP（まず動かす）

- **資源の所持・消費**：現行の UserInventory + Item で「スタック型」を継続。工業受け取り・アイテムクラフト入力での増減。
- **アイテムクラフトメニュー**：装備・メカパーツ・探索用消耗品を作る専用メニューを実装。資源・材料を消費してクラフト実行。工業設備とは別画面。
- **装備の製造と個体**：装備マスタ＋装備個体テーブル。クラフトで 1 個体生成。種別ごとのステ範囲内でランダム（個体差はステのみ）。**MVP では装備枠 6 種のうち 2 種程度**（例：鉄シリーズ・布シリーズ）を先行。装備は**派生戦闘ステ**（HP, MP, PATK, MATK, PDEF, MDEF, HIT, EVA, LUCK）への加算が役割で、属性耐性は持たせない（オミット）。装備の stats 合算値はそのまま派生戦闘ステに加算する（spec/071）。
- **装備の装着**：主人公・仲間の装備枠 6 種（主武器・副武器・頭・胴・腕・脚）＋遺物 4 枠。中間テーブルで着脱。戦闘計算では装備補正を**派生戦闘ステ（二層解釈後の HP, MP, PATK, MATK, PDEF, MDEF, HIT, EVA, LUCK）への加算**として反映する。属性耐性は遺物のみ。
- **メカパーツの製造と個体**：パーツ種別マスタ＋パーツ個体テーブル。クラフトで 1 個体生成。既存 MechaEquipment を個体参照に拡張。個体差はステのみ。属性耐性は 044 策定通り（未実装なら仕様のみ先行可）。
- **探索用消耗品**：クラフトで作成可能にし、探索で消費する流れを MVP で用意するかは実装判断（種別・効果を最低限定義）。
- **遺物**：MVP では「枠と効果表示」まででも可。遺物個体テーブルと簡易ドロップを入れる場合は、パッシブ効果テーブルを用意。
- **スキル分析書**：分析書ドロップ・消費・習得/レベル。CharacterSkill.level と「レベルアップ必要冊数」ルールを固定。
- **バッグ画面**：所持一覧を**種別ごとのタブ**で見られるようにする（資源・装備・メカパーツ・遺物・消耗品・課金など）。消費は各機能画面で行うため、バッグでは一覧確認が主。

### 4.1.1 所持上限（maxOwnedPerUser）

- **方針**: 原則として、**すべてのスタック型アイテムに「ユーザーごとの所持上限」を設ける**。
  - 型: Item 側に `maxOwnedPerUser: Int?` を持たせる（NULL なら「無制限」）。
  - 基本探索キットは **1 日 1,000 個の排出**・**1 回探索の基本消費を 10 個想定**（100 回分/日）なので、**約 3 日分を貯められる 3,000 個**を上限とするイメージ。
  - 石材・木材などの汎用素材も含め、**デフォルトの所持上限を 30,000 個/ユーザー**に揃えておき、特殊なレアアイテムだけ個別に小さな上限を設定する。
- **付与時のルール**:
  - 工業受け取り（019）、探索報酬（049）、クエスト報酬（054）、クラフト出力（046）など、**UserInventory を増やすすべての処理**は、
    - `max = item.maxOwnedPerUser ?? Infinity`
    - `newQty = min(currentQty + delta, max)`
    のように **上限でクリップ**する。  
  - `currentQty + delta` が max を超えていた場合、**超えたぶんは「破棄」扱い**とし、必要に応じて UI やログで「所持上限に達しているため一部が破棄されました」などのメッセージを出す。
- **狙い**:
  - 故意・バグ問わず、**int の上限を超えるような異常な所持量**をシステム側で先回りして防ぐ。
  - 探索資源や素材を「無限に貯めておく」ことを避け、**定期的にログインして消費・活用してもらう**プレイサイクルを維持する。

### 4.2 Next（MVP 後）

- **装備メニュー・メカ装備メニュー**：一覧・比較・ソート・フィルタなど UX を充実。装備枠 6 種の残り種別を追加。
- **遺物の生成ルール詳細**：型別パッシブ効果テーブル・エリア別ドロップテーブル・基礎ステ％補正の抽選ルールを正式 spec 化し実装。
- **スキルレベルの効果**：倍率補正・MP 軽減の数値は **spec/052 §4.3** に定義済み。物理・魔法は L0→1.00倍/L99→1.25倍・消費MP 1.0→0.9倍（sqrt）。補助は消費MP 1.0→0.8倍（sqrt）。実装は戦闘ループで CharacterSkill.level を参照して適用。
- **アイテムクラフト**：レシピ追加・UI 改善。

### 4.3 Later（後回し）

- **課金アイテム**：雇用権利書・設備加速券など。各機能画面での消費フローを spec で定義してから実装。
- **市場・取引**：装備個体・遺物個体の取引は Later。

## 5. 評価・是正提案・ヒアリング

### 5.1 評価（良い点）

- **資源／装備／メカパーツ／遺物／スキル分析書／課金** の切り分けが明確で、既存の 021・02・09・017 と矛盾しない。
- 装備・メカパーツを「個体別ユニーク」にすることで、繰り返し製作・厳選のサイクル（021）と一致している。
- 遺物の「型×エリアのドロップテーブル」「特殊スキル＋基礎ステ％2つ」は、ハクスラとして拡張しやすい構造。
- スキル分析書の「初回習得→レベルごとに必要冊数」はシンプルで理解しやすい。
- 課金を「権利書」「加速券」などアイテム化する方針は、のちの課金設計と相性がよい。

### 5.2 用語・統一

- **「バッグ」と「所持」**：「バッグ」＝UI 上の持ち物一覧（種別タブで一覧）。「所持」＝データ上のユーザー所有。
- **資源の「レア」**：017 の希少部品は Item の一種。Item に category（material_common / material_rare / consumable 等）を導入し、クラフトレシピで「material_rare を N 個」のように参照する。
- **スキル分析書の必要冊数**：「Lv0→1 に 1 冊、Lv1→2 に 2 冊」のように、レベルアップごとに必要冊数を仕様で固定する（実装時に明文化）。

### 5.3 未確定・今後の検討

- **探索用消耗品の効果一覧**：回復・バフ・ダメージ等、種別と効果を spec で列挙する。
- **装備枠 6 種のスロット定数**：主武器・副武器・頭・胴・腕・脚をコード上 enum または定数で定義する際の名前を spec で確定する。

### 5.4 他に所持が必要になりそうなアイテム（将来検討）

設計図・クラフトレシピ解放は **§2.7 設計図** として採用済み（研究メニューで消費、同じ持ち方で統一）。以下はそれ以外の候補メモ。

| 候補 | 概要 | 所持の要否・メモ |
|------|------|------------------|
| **キーアイテム** | ストーリー・クエスト用（エリア解放の鍵、依頼品など）。通常はスタックしない・売却不可。 | 進行系をアイテムで表現する場合。Later で検討。 |
| **交換用メダル・イベントポイント** | イベント報酬で入手し、専用交換所で景品と交換。 | 通貨で持つか、Item（category=exchange_medal）で所持するか。所持するならスタック。 |
| **強化素材** | 装備・メカパーツの「強化」に消費（例：ステ加算・やり直し）。 | 現状は個体差は製造時のみで強化なし。将来やるならクラフト/装備画面で消費するスタック品。 |
| **仲間解放系** | 雇用権利書とは別に「この仲間を雇用可能にする」ドロップ（例：〇〇の契約書）。 | 課金以外の解放経路を用意するなら。消費で「その仲間を選択可能にする」など。 |

## 6. 次のステップ

1. **データモデル・spec**：  
   - 資源：現行 UserInventory + Item のまま、Item に category 追加を検討。  
   - **アイテムクラフト**：クラフトレシピ（入力資源・出力＝装備種別/パーツ種別/消耗品）と実行 API を spec に起こす。工業設備の Recipe とは別体系。  
   - 装備：EquipmentType（マスタ）・EquipmentInstance（個体）・CharacterEquipment（装着、スロット＝主武器/副武器/頭/胴/腕/脚）のテーブル設計を spec に起こす。  
   - メカパーツ：MechaPartType は 044 のまま。MechaPartInstance（個体）を追加し、MechaEquipment から instance を参照する形に拡張する spec を書く。  
   - 遺物：RelicType（型）・遺物用パッシブ効果テーブル・RelicInstance（個体）・ドロップテーブル・装着（CharacterRelic 等、4 枠）を spec で定義する。  
   - 探索用消耗品：Item の一種（category=consumable 等）＋探索中の消費 API。  
   - スキル分析書：Item の一種＋消費時アクション（習得/レベルアップ）。CharacterSkill.level と必要冊数ルールを spec に書く。
   - **設計図**：Item（category=blueprint 等）＋UserInventory。研究メニューで消費し、解放対象（設備／設備型／クラフトレシピ）をマスタで紐づける。研究 spec で「消費 API・解放結果」を定義する。
2. **バッグ UI**：種別ごとのタブ（資源・装備・メカパーツ・遺物・消耗品・課金など）で一覧。消費は各機能画面で行う。装備との連携は装備メニュー側で実施。
3. **AGENTS.md**：アイテム・所持・アイテムクラフトに関する spec と実装場所の対応が決まり次第、AGENTS.md の「Spec ↔ 実装」に追加する。

## 7. 拡張耐性（今後増やしたいとき）

以下のようにしておけば、追加時も既存設計を崩しにくい。

| 拡張したいもの | 方針 | 備考 |
|----------------|------|------|
| **新規アイテム種別** | Item の **category**（または種別テーブル）で区別。消費先は「この種別はどこで消費するか」をマスタまたはコードで紐づける。 | キーアイテム・交換メダル・強化素材などは category 追加＋消費 API の拡張で対応。バッグのタブも category 一覧から出せば増減しやすい。 |
| **新規解放対象（設計図）** | 設計図アイテムに「解放対象タイプ＋対象ID」をマスタで持たせる。研究の消費 API は「対象タイプ」で分岐する。 | 設備・設備型・クラフトレシピのほか、メカパーツ種別解放・探索エリア解放なども同じ枠で追加可能。 |
| **建設レシピの資源** | 型ごとの建設レシピを「入力 ItemID＋数量」のリストで持つ。新規資源（Item）を追加したら、レシピに登録するだけ。 | 017。希少部品必須廃止により、どんな資源の組み合わせでもレシピ定義可能。 |
| **装備枠・遺物枠の数** | スロット種別は **enum またはマスタ**で定義。6 種・遺物 4 は現時点の想定。増やす場合はスロット定数追加と CharacterEquipment / CharacterRelic のスロット値の許容範囲を拡張。 | 仕様で「スロット種別はマスタで管理し、コードはその一覧を参照する」にしておくと変更しやすい。 |
| **クラフトレシピ** | レシピ＝入力（ItemID＋数量）のリスト＋出力（装備種別／パーツ種別／ItemID 等）。新規装備種別・新規消耗品は出力側のマスタを増やすだけ。 | 工業設備の Recipe と別体系のため、クラフト側だけの拡張で済む。 |
| **課金アイテムの効果** | アイテムマスタに「効果種別＋効果用パラメータ」を持たせる。消費は各機能画面で行うため、新効果を足すときは「効果種別の分岐」と「その効果を使う画面」を 1 つずつ追加。 | 効果一覧を spec または定数で列挙し、消費時のルーティングを効果種別で振り分ける形にしておく。 |

**データ側の心がけ**  
- 種別・スロット・解放対象・効果は**マスタまたは enum で列挙**し、ハードコードを減らす。  
- 「〇〇の一覧」は DB のマスタテーブルまたは設定から取得し、コード上で固定リストにしない。  
- 新規種別追加時は、026 の一覧表と該当 §2.x に 1 行ずつ追記し、必要なら spec を更新してから実装する。

## 9. Spec 化前の確定事項（実装 Spec 用）

Spec 045・046・047 に起こす際の前提。ヒアリングで確定した内容をまとめたもの。

### 建設（017）

- **A-1** 基本型も「その型の建設レシピで定義した資源を消費して建てる」。無料ではない。
- **A-2** 1 型 : 1 レシピ。作れる成果物は同じでも、時間・素材効率・排出量などが型で変わる。それらをまとめてレシピとみなし、型ごとに 1 レシピを持つ。

### 設計図・解放（026）

- **B-1** 1 設計図アイテム = 1 解放対象で固定。
- **B-2** 解放状態は**テーブルを分けて管理**する（Cursor の実装・保守がしやすい方）。  
  - 例：`UserFacilityTypeUnlock(userId, facilityTypeId)`  
  - 例：`UserFacilityVariantUnlock(userId, variantId)`  
  - 例：`UserCraftRecipeUnlock(userId, craftRecipeId)`  

### アイテム・装備

- **C-1** Item.category は `material` / `consumable` / `blueprint` / `skill_book` / `paid` 等で揃える。装備・メカパーツ・遺物は個体で別テーブル。
- **C-2** 装備スロットのコード値は `main_weapon` / `sub_weapon` / `head` / `body` / `arm` / `leg` で統一。

### 研究メニュー

- **D-1** 「研究」で (1) 設備の基本型解放（研究条件達成）と (2) 設計図消費（設備型・クラフトレシピ解放）の両方を行う。

### Spec 分割・MVP

- **E-1** Spec は 045（アイテム・所持・バッグ）、046（アイテムクラフト）、047（研究・解放・建設）に分割する。
- **E-2** MVP スコープは**上から 3 つ**：(1) 資源の所持・消費（既存＋建設での消費）、(2) 建設レシピ＋設備配置時の資源消費、(3) クラフト＋装備個体まで。**派生型の解放（設計図で設備型を解放）は後回し**。

### メカパーツ（044）

- **F-1** 個体管理する場合は `MechaPartInstance` を新設し、`MechaEquipment` を `mechaPartInstanceId` 参照に変更する（装備の EquipmentInstance + CharacterEquipment と同じパターン）。
