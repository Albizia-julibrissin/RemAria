# キャラ周り実装の検討事項

Character（主人公・仲間・メカ）実装にあたり、決めておくべき項目の洗い出し。  
`docs/02_domain_model.md`・`05_project_constitution.md`・`04_persistence_principles.md` を前提とする。

---

## 1. ステータス 【決定】

### 1.1 永続化する項目（Character にカラム）

| 項目 | 用途 | 備考 |
|------|------|------|
| **STR**（筋力） | 戦闘・訓練 | 訓練で変動 |
| **INT**（知力） | 戦闘・訓練 | 同上 |
| **DEX**（敏捷） | 戦闘 | - |
| **VIT**（耐久） | 戦闘・HP 算出等 | - |
| **SPD** | 行動順・手数 | 戦闘で必須 |
| **CAP**（総合上限） | 育成天井 | 憲法：レベル非採用・CAP採用 |
| **LUK**（運） | クリティカル率等 | 実装する |

- **結論**：**STR / INT / DEX / VIT / SPD / CAP / LUK** を Character にカラムで持つ。**中間テーブルは使わない**（全キャラとも同じステータス種別）。
- **HP / MP**：保存しない。**STR 等から計算して出す**。戦闘計算や表示時に都度算出する。
- **永続化**（04）：現在の**基礎ステータス**と **CAP** のみ保存。装備・バフ込みの最終値も保存しない。
- メカはパーツで性能が変わる想定のため、**基礎ステータスをパーツから算出**するか、Character にキャッシュするかは別途検討。

### 1.3 訓練との関係

- 訓練 Job 完了時に「どのキャラのどのステータスをいくつ変動させるか」を確定し、Character の該当カラムを更新。
- 変動後も「合計が CAP を超えない」などのルールをどこで保証するか（Job 完了処理内でチェック等）。

## 2. スキル 【方針決定】

- **スキルマスタ**（スキル定義）と **Character** を **習得スキルテーブル**（例: CharacterSkill）で連関させる。キャラが持つスキルは「習得スキル」として中間テーブルで管理する。

### 2.1 種別（ドメイン準拠）

| 種別 | 主人公 | 仲間 | メカ |
|------|--------|------|------|
| 戦闘アクティブ | ○ | ○ | ○（同一参照） |
| 戦闘パッシブ | ○ | ○ | ○ |
| 工業パッシブ | 原則なし | ○（厳選） | - |
| 才能パッシブ | - | ○（CAP到達で獲得） | - |
| 主人公専用効果 | ○（Character のカラム） | - | - |

### 2.2 設計

- **マスタ**：スキル定義テーブル（Skill 等）。スキルID・名前・効果種別・消費MP 等。
- **習得**：**習得スキルテーブル**（CharacterSkill：characterId, skillId, 習得日等）。スキルテーブルとキャラクターテーブルを連関させる。
- **戦闘での参照**：パーティ3体の Character を渡し、習得スキルを同じ手順で参照。category で分岐するのは「工業パッシブの有無」など必要最小限。

## 3. 装備・遺物 【方針】

### 3.1 中間テーブル vs カラム（装備の観点）

- **中間テーブル**（CharacterEquipment 等）：キャラ×スロット×装備個体を行で持つ。スロット種別の追加がマスタや enum の拡張で済む。装備枠数や種別を後から変えやすい。
- **カラム**（Character に weaponId, armorId 等）：シンプルだが、枠を増やすたびに ALTER。枠が固定なら可。
- **結論**：装備はスキルと同様、**キャラとアイテム（装備個体）を連関させる中間テーブル**（スロット種別を持つ）で持つ形が拡張しやすい。

### 3.2 装備

- **装備枠**：武器／防具など。Character に紐づく「装着」を中間テーブル（例: CharacterEquipment）で管理（characterId, slotType, equipmentInstanceId）。
- **基礎値反映**：戦闘計算時に「キャラ基礎＋装備」で都度計算。永続化しない（04）。

### 3.3 遺物 【後回し】

- **遺物**：ハクスラ要素でアイテムごとにユニーク。4枠の仕様は 01_features・憲法にあるが、設計は**いったん後回し**。あとでちゃんと検討する。

## 4. パーティ・出撃 【決定】

### 4.1 固定3枠（憲法）

1. 主人公 1体（User.protagonistCharacterId）
2. 仲間 1体
3. メカ 1体

### 4.2 設計

- **パーティプリセット**を用意し、そこに「どのキャラを編成しているか」を覚えさせる。
- テーブル案：**PartyPreset**（または Party）— userId, presetName, slot1CharacterId（主人公枠）, slot2CharacterId（仲間枠）, slot3CharacterId（メカ枠）、など。複数プリセット持つなら presetIndex や name を追加。
- 出撃時：選択中のプリセットから3体の Character id を戦闘入力に渡す。戦闘ロジックは Character のステータス・スキルを同じように参照。

## 5. キャラの作成・入手 【方針】

- **作成・入手方法の詳細はまだ未定**。以下は初期状態の方針。

### 5.1 主人公

- **必ず1体**：アカウント作成時（または初回ログイン時）に Character（category=protagonist）を1体作成し、User.protagonistCharacterId に設定。
- 初期ステータス・初期スキルをどこで定義するか（定数／マスタ／初期化処理）は別途決定。

### 5.2 仲間

- **初期**：**仲間が3人いる、もしくは入手できる状態**でスタートする。所持3体 or 初期で3体獲得可能など、詳細は未定。

### 5.3 メカ

- **初期**：**メカは1体を初期配備**する。

---

## 6. 訓練との接続 【決定】

- **Assignment**：設備（FacilityInstance）に「どのキャラを配置したか」を割り当て。ドメインでは本人は研究室常駐で Assignment 持たない（MVP）。
- **Job**：訓練 Job は「対象キャラID ＋ 設備 ＋ startAt/finishAt」。完了時にそのキャラのステータスを変動させる。
- **対象**：**設備への配備は主人公と仲間のみ。メカは無し。**

---

## 7. メカのパーツ 【決定】

- **装備と同じくスロットで部位分けして装備**する。頭・胴・腕など部位ごとにスロットを用意し、パーツ個体を装着。
- **パーツそのものにスキルがついている**。例：特定の頭パーツを装備したらヒールが使える、など。装着したパーツが持つスキルを、そのメカの「使えるスキル」として参照する。
- **MechPart**：メカ（Character category=mech）に紐づく「装着情報」。部位スロット×パーツ個体。パーツマスタにはスキル付与やステ補正を持たせる想定。

---

## 8. 表示・識別 【決定】

### 8.1 表示名

- **キャラごとに1つ**。Character.displayName で持つ。
- **変更方法**はあとで考える（誰が・どこで変えられるかは未定）。

### 8.2 見た目（アイコン）

- **1キャラ1アイコン**。
- **アイコンのファイル名を保存**する。ファイル名はすべて数値の予定（例: `1.gif`, `42.gif`）なので、その文字列を Character に持たせる（例: iconFilename。参照パスは `/icons/{iconFilename}` など）。

### 8.3 並び・種別

- **並び**：未定。
- **種別**（ラベル等）：未定。

### 8.4 説明文

- **持っておく**。Character に説明文用のカラム（例: description, テキスト NULL可）を用意する。
- 仲間キャラ詳細画面などで表示する想定。**画面仕様はまだ未定**。

### 8.5 非表示・ロック

- **無し**。非表示やロックの条件は設けない。
