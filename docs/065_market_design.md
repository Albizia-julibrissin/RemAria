# 市場設計（正式）

市場機能の設計・仕様の正本。実装時は本 doc と **spec/** の市場 spec を参照する。

**由来**: `docs/ideas/market.md` の内容を一般 doc に移し、運営設定・購入ルール・一般的なゲームで必要な項目を追加して仕様を詰めた。

---

## 1. 目的・スコープ

- **プレイヤー間のアイテム売買**をゲーム通貨で行う市場を提供する。
- 探索用資源（基本探索キット）の余剰・不足を取引で吸収し、ゲームサイクル（`docs/064_game_cycle_design.md`）の「延々派・適度派」の折り合いを取る。
- 無課金の通貨調達経路の一つとして「市場でレア等を売る」を位置づける。
- **実質的なトレード（特定プレイヤー間の直接取引）は防止**し、市場経由の匿名性の高い取引に限定する。

---

## 2. 前提・他 doc との関係

| 項目 | 方針 |
|------|------|
| 通貨 | **GRA**（`User.premiumCurrencyFreeBalance` / `premiumCurrencyPaidBalance`）。購入時は無償から消費。売り手入金はすべて無償 GRA（docs/076）。 |
| NPC | **NPC 販売・買取は行わない**。価格はプレイヤー間取引のみ。 |
| 取引対象 | **スタック型**（資源・消耗品・設計図・スキル分析書）はマスタで「市場出品可」とした Item のみ。**遺物**は本体を 1 個ずつ出品・購入可（§8）。**装備・メカパーツ・特別（課金）**は出品不可。 |
| 参照 | 機能一覧 `docs/01_features.md`（出品/購入・手数料・価格履歴）、希少部品 `docs/017_facility_variant_and_rare_parts.md`、ゲームサイクル `docs/064_game_cycle_design.md`。 |

- **機能解放**: 市場は**特定の開拓任務を完了するまで利用不可**とする。**ユーザーごとに解放フラグ**（例: User.marketUnlocked）を持たせ、任務側でそのフラグをオンにする処理を後から追加する。市場 API はフラグ未設定ならエラーを返す。

## 3. アイテム側の設定（運営の取捨選択）

**「流してよいもの／流したくないもの」を運営が取捨選択できるようにする。**

### 3.1 出品可否（Item マスタ）

- **Item に「市場出品可否」を持たせる。**
  - 例: `marketListable`（Boolean, デフォルト false）。**true のアイテムのみ**出品可能とする。
- 運営はマスタ編集で「基本探索キット」「希少部品（レア資源）」など**出品を許可したいアイテムだけ**を true にし、それ以外（課金アイテム・スキル分析書・特定の素材など）は false のままにできる。
- 将来的に装備部品・遺物欠片を Item として扱う場合も、その Item の `marketListable` で制御する。

### 3.2 アイテム別の価格・数量制約（任意）

- **グローバル定数**で「最低価格」「最低出品単位」の NULL 時フォールバックを持ち、**Item ごとにマスタで上書き可能**にしておくと運用しやすい。
  - 例: `marketMinPricePerUnit`（Int?, NULL ならグローバル定数を使用）、`marketMinQuantity`（Int?, NULL ならグローバル定数を使用）。
- レア資源だけ最低価格を高めにする、探索キットだけ最低出品単位を 10 にする、といった**品目別チューニング**が可能になる。

### 3.3 データ（スキーマ方向）

- Item に以下を追加する想定（正式な型・名前は spec および schema で定義）。
  - `marketListable`: Boolean, default false。
  - （任意）`marketMinPricePerUnit`: Int?, NULL ならグローバル定数を使用。
  - （任意）`marketMinQuantity`: Int?, NULL ならグローバル定数を使用。

## 4. 購入ルール：最安から購入（実質トレード防止）

**「特定の出品者から買う」を選べないようにし、実質的なプレイヤー間トレードを防ぐ。**

### 4.1 ルール

- 購入時、**同一アイテムの「最安の出品」から自動で消化**する。
  - 購入者は「誰の出品を買うか」を指定できない。**常に最安から買う**。
- 同額最安が複数ある場合は、**出品の古い順（先に出したものから）** で消化する等、仕様で一意に決める（spec で定義）。
- **1 つの出品（listing）は数量（quantity）を持つ**。購入のたびにその listing の quantity を減らす（**部分消化・partially fill**）。quantity が 0 になった listing は削除する。同一 listing が複数回の成約で少しずつ減っていく形になる。
- 結果として「A が B にだけ売る」「B が A からだけ買う」という**直接取引が成立しない**。RMT や不正な物々交換の温床になりにくい。

### 4.2 売り手の匿名性

- **購入画面では売り手（出品者）を表示しない**。成約後も「○○さんから購入」のような表示は行わない、または最小限にする。
- 成約履歴で「誰に売れたか」を売り手に通知するかどうかは運用方針次第（通知する場合は「プレイヤー名」ではなく「成約した」事実のみなど、匿名性を保つ選択肢もある）。

## 5. 出品・手数料・価格制約（複数アカウント対策含む）

| 項目 | 内容 |
|------|------|
| **手数料** | **売り手から 10％** を取る。成約時に売り手の受け取りから控除。**端数は切り捨て**。買い手には手数料なし。 |
| **最低価格** | 出品には**単価の下限**を設ける。**アイテムごとに Item マスタ**（marketMinPricePerUnit）で持ち、管理画面で編集可能。NULL のときは**グローバル定数**を使用。 |
| **最低出品単位** | 出品は**一定数量以上**に限定する。**アイテムごとに Item マスタ**（marketMinQuantity）で持ち、管理画面で編集可能。NULL のときは**グローバル定数**を使用。 |
| **上限価格（任意）** | 単価の**上限**を設けると、高額転売・価格つり上げの抑止になる。実装は Phase 2 でも可。 |

## 6. その他一般的に取り込む項目

以下のように整理する。実装 Phase は spec で切る。

### 6.1 出品まわり

| 項目 | 内容 |
|------|------|
| **出品の有効期限** | 出品は「○日間有効」とし、**有効日数は定数**で持つ（例: 7 日）。期限切れになった listing は**自動で取り下げ**（在庫をバッグに戻し、listing を削除）。自動取下げが発生した旨は**履歴**に記録し、プレイヤーは「履歴」タブで確認できる。 |
| **同時出品数上限** | 1 ユーザーあたりの有効な listing 件数上限。**プレイヤーごとに変えられる**ようにする（将来サブスクで「通常10出品まで → サブスク20出品まで」とする想定）。**通常用とサブスク用の2値を定数**で持ち（例: 通常 10、サブスク 20）、ユーザーのサブスク状態に応じてどちらを使うか切り替える。サブスク未実装時は通常の定数のみ使用。値の変更をデプロイなしで行いたい場合は、のちに設定テーブルやプランマスタへ移す。 |
| **出品キャンセル** | **売り手は出品を取り下げ可能**。取り下げ時は在庫を即時戻す。成約前なら手数料は発生しない。 |

### 6.2 購入まわり

- **買い注文（買いオーダー）は設けない**。**売り注文（出品）のみ**でシンプルに保つ。購入は「現在の最安出品を買う」のみ。

**購入処理のトランザクションと競合対策**

- B と C が同時に同一アイテムの最安 listing を購入した場合、在庫の二重消化や通貨の不整合が起きないよう、以下を仕様とする。
- **購入処理は 1 つの DB トランザクション内で完結させる**。その中で「買い手の通貨減算」「listing の quantity 減少」「売り手への通貨加算（手数料控除後）」「成約ログ（MarketTransaction）の挿入」を一括でコミットする（**購入時に即・通貨を確保**する形でよい）。
- 最安 listing を更新する際は、その行を**ロック**する（例: `SELECT ... FOR UPDATE`）。同時に 2 リクエストが同じ listing を読んで両方とも quantity を減らしてしまうことを防ぎ、部分消化が一意に決まるようにする。

### 6.3 価格履歴・表示

| 項目 | 内容 |
|------|------|
| **価格履歴** | 直近の成約価格（例: 直近 N 件の平均または中央値）を**参考表示**する。`docs/01_features.md` の「価格履歴」に相当。出品・購入の判断材料とし、乱高下の抑制にもつなげる。 |
| **価格履歴の実装** | **成約ログテーブル 1 本**で足りる。例: **MarketTransaction**（id, itemId, pricePerUnit, quantity, timestamp）。価格履歴の表示はこのテーブルのみを参照し、**UI 側で**直近 N 件の **AVG / MEDIAN / MIN / MAX** などを算出して表示する。別途「価格履歴専用テーブル」は設けない。 |
| **現在の最安・在庫** | 一覧では「そのアイテムの現在の最安単価」と「その価格で買える数量」を表示。購入は「数量」を指定すると最安から順に消化。 |

### 6.4 その他

| 項目 | 内容 |
|------|------|
| **スタック** | 同一アイテムはスタック型なので、1 出品で「数量」をまとめて出品できる。購入時は「○個買う」と指定し、最安の出品から順に消化。 |
| **通貨の流れ** | 成約時：買い手の GRA を減算（無償→有償の順）。売り手には手数料控除後を premiumCurrencyFreeBalance に加算。手数料分はシステムに吸収（docs/076）。 |

## 7. 経済設計・複数アカウント対策（要約）

- **通貨を産まない**: 通常ループ（探索・生産・戦闘）ではゲーム通貨が増えない設計。通貨の供給は**一過性**（クエスト報酬・初回クリア・実績等のごく少量）のみ。
- **デフレ寄せ**: 通貨は基本的に流出のみ。市場手数料も「消える」だけなのでデフレに寄る。
- **複垢対策**: 手数料 10％・最低価格・最低出品単位で、サブ垢が安く大量に流し込むメリットを抑える。加えて**最安から購入**にすることで、サブ垢→メインへの「特定間でだけ成約」が成立しにくい。
- **無課金の通貨**: (1) 市場でレア等を売る、(2) クエスト等の一過性少量。課金者は市場で資源を買って回す or 課金で通貨を増やす。

詳細は旧 `docs/ideas/market.md` の「経済設計」「無課金の通貨調達」を参照。本 doc が正本のため、矛盾がある場合は本 doc に合わせる。

## 8. 取引対象の種別・装備・遺物の扱い

### 8.1 売れるもの・売れないもの

- **売れるもの**: **資源・消耗品・設計図・スキル分析書**（Item のスタック型。UserInventory の数量で扱う）、**遺物**（RelicInstance を 1 個ずつ。データ・画面がスタック型と異なる）。
- **売れないもの**: **特別・課金**（Item.category=paid 等は marketListable=false）、**装備**（EquipmentInstance。市場では扱わない。将来は部品のみ出品可）、**メカパーツ**（MechaPartInstance。市場では扱わない）。
- スタック型は在庫数の増減で出品・購入が完結する。**遺物は 1 個ずつステータスが違う**ため、出品は「この RelicInstance を出品」、購入は「この 1 個を購入」で、見せ方と処理がスタック型と別枠になる。詳細は spec/075 §8。

### 8.2 装備・遺物の補足

- **装備**: 本体の出品は不可。将来は解体で部品に戻し、その部品を Item として `marketListable` で出品可にする。
- **遺物**: **本体を 1 個ずつ**出品・購入可能とする。欠片（スタック型 Item）も別途 marketListable で出品可にできる。解体・部品・欠片の詳細は別 spec に委ねる。

## 9. Spec・実装への落とし方

- **市場の正式 API・データ構造・実装フェーズ・画面構成**は **`spec/075_market.md`** に定義する。
  - 出品 API（listMarketItem）、購入 API（buyFromMarket・1 トランザクション・部分消化・行ロック）、出品取り下げ（cancelMarketListing）、一覧取得（getMarketList）、価格履歴（getMarketPriceHistory）、履歴（getMarketUserHistory）。
  - **実装フェーズ**は spec 075 の **§7** に記載。Phase 1（最小限の出品・購入・一覧・管理画面）、Phase 2（価格履歴・同時出品数上限・有効期限・**自動取下げと履歴**・上限価格・Item 別制約）、Phase 3（運用・UX 調整）に分けている。
  - **画面の構成・遷移**は spec 075 の **§9** に記載。**購入・出品・取下げ・履歴**の 4 つのボタンで分岐し、それぞれ別画面で構成する。
  - **取引対象の種別**（売れるもの・売れないもの）と**購入側の画面・機能**（スタック型は数量・在庫変動、遺物は 1 個ずつ・見せ方と処理が別枠）は spec 075 の **§8** に記載。
- **DB**: MarketListing（出品）、MarketTransaction（成約ログ・価格履歴の元データとして 1 本で運用。§6.3）。Item の拡張（§3.3）は `prisma/schema.prisma` と `docs/08_database_schema.md` に反映する。
- **MarketListing のインデックス**: 最安取得クエリ（`ORDER BY pricePerUnit, createdAt LIMIT ...`）を高速化するため、**複合インデックス (itemId, pricePerUnit, createdAt)** を定義する。購入時に毎回使うため必須とする。
- **管理画面**: Item マスタ編集で `marketListable`（と任意で marketMinPricePerUnit / marketMinQuantity）を編集できるようにする。`manage/admin_item_master_edit.md` と整合させる。
